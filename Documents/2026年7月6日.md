## 1. GAPBSクラッシュの根本原因特定と修正

前日までの調査を継続。GAPBS(BFS/PR)がSID上でsendInstruction系SIGSEGV(signal 11)で
クラッシュする問題を深追い。

**根本原因**: `sift/recorder/recorder_base.cc`の`sendInstruction()`内、`dyn_addresses`への
生ポインタタッチが、Pinの`PIN_AddInternalExceptionHandler`(内部例外ハンドラ)にキャッチされ、
POSIXシグナルハンドラには一切届かない設計だったため、当初実装した signal+sigsetjmp ベースの
タッチガードは最初から機能していなかったと判明。

**修正**: 生ポインタの直接タッチを、Pin公式の安全メモリアクセスAPI`PIN_SafeCopy()`
(無効アドレスなら0バイトコピーでクラッシュせず返る)に置き換え。BFS/PR×2TH/8THの
スモークテストで全コア正常な命令数(非ゼロ)を確認し、`docker.io/snipersim/snipersim:latest`
に反映(旧ロック混入版は`:pre-wake-fix`→さらに後で`:with-broken-lock-fix`等のタグで保存)。

途中、「Sift::Writerのクローズ処理を排他ロックで囲む」という別の修正案も試したが、
これはPageRank(PR)の記録を壊す(Core0のみ記録、他コア0命令)副作用が判明し破棄。
最終的に採用したのはPIN_SafeCopyのみのシンプルな修正。

「8THは7/8スレッド完了ってどういうことやねん」というユーザーの指摘を受け、中途半端な
「ほぼ解決」という報告を撤回し、BFS/PR×8/12/16THでのデータ完全性検証を継続。最終的に
SafeCopy修正版でBFS 8THの全8コアが2.6M〜11M命令という妥当な分布で記録されることを確認。

## 5. WATER_NSQUAREDの深追い: Sniper本体のfutexデッドロック発見

WATER_NSQUARED(SPLASH2)が8THで確率的にハングする問題を深追い。gdbでアタッチしたところ、
**全8スレッドが完全に同一の命令アドレスで停止**していることを確認(単発の遅延ではなく
真性の相互デッドロック)。6並列実行で2/6がハング、単独実行では6/6成功という
タイミング依存の再現パターン。

Sniper本体のソース(`common/system/syscall_server.cc`)を調査し、`FUTEX_CMP_REQUEUE`
実装に開発者自身の`"race condition may have occured"`というコメントを発見。各コアが
独立したホストスレッドでトレースを再生する設計上、実行時の本当のhappens-before関係を
SIFTトレース形式が保持していないため、記録時と異なる順序でFUTEX_WAIT/WAKEが処理され、
永遠に来ないWAKEを待つスレッドが生まれうると根本原因を特定。

`TraceManager::signalDone()`に「1スレッドだけ残ったら強制resumeThread()する」修正を
追加し本番`:latest`に反映(fluidanimateで実際に発生していた「最後の1スレッドが
siblingの終了を待ち続ける」パターンには有効)。しかしWATER_NSQUARED/BFSで見られる
「全スレッド同時凍結」パターンには無効(誰もsignalDone()を呼ばないため)。真の修正には
シミュレータのコア部分(トレースリプレイの順序保証)の再設計が必要と判断し、深追いを断念。

WATER_NSQUARED、および後に同じ症状を再現したBFS/PR(GAPBS)を`WORKLOADS`から除外。

## 14. 教授の助言: 数式(既知の物理)+AIの残差学習というハイブリッド構成

指導教員から「全部MLに任せず、既知の相関は数式で決め打ちし、未知の相関だけAIに学習させる方が
良い」との助言。7/5に「AIによる自動最適化の一貫性を欠く」として見送っていたルーフライン
輻輳モデルが、実は**ワークロード非依存の一般物理法則**(手作りの憶測ではない)である以上、
この助言と矛盾しないと整理。

### ルーフラインモデルの設計
既存の`akarin/cpsat_mapper.py`は`alpha*makespan + (1-alpha)*remote_penalty`という、
**単位が違う2つの量(秒 vs 無次元の通信量)をalphaで無理やり混合**していた。これを
`node_finish[n] = max(compute_bound[n], bandwidth_bound[n])`という、両方とも「秒」に
統一した式に置き換える設計を提案:
- `compute_bound[n]`: ノード内の一番遅いスレッドの実行時間(既存のmakespan相当)
- `bandwidth_bound[n]`(新規): ノードに乗った全スレッドの合計メモリアクセス量(comm_size.csv、
  実バイト)÷実メモリ帯域(51.2GB/s、config/generate_config.py既知の値)
- alphaという恣意的パラメータを排除

### 実データでの検証: 「別ノード通信の罰」ではなく「同ノード帯域競合の罰」が正解
SP_W/2THの5戦略の実測cpu_map+実測sim_timeで検証したところ、当初モデル化していた
「別ノード間の通信コスト」ではなく、**「同ノード内で全スレッドが同じメモリコントローラの
帯域を奪い合う」という競合モデル**が正解だと判明。旧remote_penaltyはこのケースで完全に
符号が逆(Scatter/MPOを「別ノードだから悪い」と評価するが実際は最速)。修正後、新
bandwidth_boundは上位2戦略(Scatter/MPO)を正確に的中。

NPB全体(BT/FT/IS/MG/CG/SP、258件、48グループ)で検証したところ:
```
旧remoteのみ相関(平均): -0.438(実測と逆相関)
新bandwidthのみ相関(平均): +0.339(正相関)
最速戦略の的中率: 旧8.3%(ランダム20%より悪い) / 新20.8%(ランダム並み)
```
数式単体では「実測と逆」から「ランダム並み」まで改善したが、まだ実用的な精度ではない
——ここがAIによる残差学習に任せるべき部分、という位置づけが実証的に裏付けられた。

### TabPFN検証: 分類は多数派クラスへの逃げ、回帰の方が誠実
既存の`MachineLearning/TabPFN/tabpfn_model.py`のtabpfn-client認証を再利用し
(新規スクリプトにトークンを複製しないよう注意)、leave-one-workload-outで検証:

- **分類(5択、1グループ=1行に5戦略の特徴量を横並び)**: 正解率44.2%と一見改善したが、
  43件中42件で常に"Scatter"を予測しており、Scatterが正解の46.5%とほぼ一致。
  **多数派クラスへの逃げに過ぎず、実質的な学習効果ではないと判明**
- **回帰(sim_time_msそのものを予測、1戦略=1行)に切替**: 個別ワークロードの相関は
  0.87〜0.99と高いが、これはスレッド数による桁違いのスケール差を拾っているだけの
  見かけ倒し。実用的な指標である「グループ単位の最速戦略的中率」は**33.3%**
  (ランダム20%・数式単体20.8%より明確に上、かつ分類のような崩壊は見られない)
- **compute_boundを命令数+実測IPCベースに直す試み**: 逆に悪化(相関-0.308、的中率8.3%)。
  P/E両方の実測IPCが同一グループ内に揃うのが16THのみ(258→12件に激減)という
  サンプル数の犠牲、およびIPC自体がその戦略固有の輻輳を反映してしまう汚染の影響
- **8THのみに絞った回帰**: さらに悪化(16.7%)。データ量(258→66件)を削る代償の方が、
  スレッド数を揃える利点より大きいと判明
- **8TH通信行列(8x8=64要素)をそのまま特徴量にした分類**: 50%(6/12)。ただし正解の
  Scatter比率も50%なので数字上はトリビアルな基準と同点。ただしIS/MGでは実際に
  HPOという非多数派の正解を的中させており、単純な多数派依存とは違う挙動も見られた
  (n=12と極小なので結論は保留)

**全体を通した教訓**: この規模のデータでは「特徴量の質を上げる/データを絞ってクリアにする」
より「サンプル数を確保する」方が効果が大きい。今夜のバッチ(8ワークロード×2クラス×
4スレッド数、AKARIN候補込み)が完了すれば、一貫性のある新しい教師データで再検証できる。

## 15. 次元の呪い打開策の文献調査: NetLSD(グラフスペクトル記述子)による検索型推定

14節の末尾で「通信行列を16×16にタイリングして特徴量を増やす」案を試したところ
的中率4.2%(2/48)と全実験中最悪の結果になり、「サンプル数(数百件)に対して特徴量が
多すぎる(257次元)」という次元の呪いが明確に確認された。この状況を打開する文献を調査。

### 調査で見つけた手法
- **NetLSD**([Spectral embedding of weighted graphs](https://arxiv.org/pdf/1910.05534)、
  [Fast Attributed Graph Embedding via Density of States](https://arxiv.org/pdf/2110.05228)):
  グラフラプラシアンの熱核トレース`h(t)=Σexp(-t*λ_i)`を複数の時間スケールでサンプリングし、
  ノード数(=スレッド数)に依存しない固定長・順列不変・サイズ不変のベクトルを作る手法。
  タイリングのような「偽の周期性の捏造」なしに8×8と16×16の通信行列を同じ次元数で
  比較できる
- **Active Subspace法**([Surrogate assisted active subspace](https://arxiv.org/pdf/2105.04979)):
  勾配情報から高次元入力の中で本当に効いている低次元部分空間だけを抽出し、そこに
  射影してから回帰する手法。N<dでも機能する設計

### NetLSDを「回帰特徴量」として使う→失敗
通信行列(20次元のNetLSD記述子)を既存の4スカラー特徴量に追加してTabPFN回帰:

```
既存4特徴量のみ(ベースライン):        相関0.576  グループ的中率33.3%(16/48)
NetLSDのみ(20次元):                  相関0.086  グループ的中率 4.2%( 2/48、ランダム以下)
既存4特徴量+NetLSD(25次元):          相関0.583  グループ的中率27.1%(13/48、悪化)
```

原因: NetLSD記述子は通信行列そのものから計算するため、**同じ(workload,class,threads)
グループ内の5戦略すべてで完全に同一の値**になる(通信行列は戦略に依存しないという
既知の事実の帰結)。つまりTabPFNから見るとグループ内で分散ゼロの次元を20個混ぜた
だけで、どの戦略が最速かを区別する情報を一切持たず、少サンプル下でモデルの注意力を
薄める純粋なノイズとして働いた。

### NetLSDを「k-NN類似度検索」に転用→成功
「回帰特徴量」ではなく「未知ワークロードが既知のどれと構造的に似ているかを探し、
その正解(実測最速戦略)をそのまま借りてくる」検索方式に切り替えたところ、
今日一連の実験の中で最良の結果になった:

```
ランダム:                    20%
数式単体argmin:              20.8%
数式回帰(4特徴量、旧ベスト):   33.3%
NetLSD検索 k=1:              39.6%
NetLSD検索 k=3〜5(多数決):    50.0%(48グループ版) / 51.2%(43グループ版)
```

「同スレッド数限定で検索」と「スレッド数不問(サイズ不変性を活用)で検索」が完全に
同じ結果になった(=NetLSD空間では同スレッド数のグループが自然と最近傍になる傾向)。
サイズ不変性は実害なし・大きな実利もなしだったが、理論通り壊れていないことは確認できた。

**解釈**: 数式(物理法則)は戦略ごとの違いを直接説明し、NetLSD検索は「未知ワークロードが
既知のどれに近いか」という教授の言う"知りようもない相関"の部分を担う——ハイブリッド
構成が意図した役割分担にちょうど一致した。

### 「数式単体」も検索型にしてみる→数式回帰と同水準止まり
5戦略のold_makespan/old_remote/new_compute/new_bandwidth(20次元)を1グループの
「数式プロファイル」とみなし、同じk-NN検索を適用:

```
数式プロファイル検索(標準化, k=1〜3): 34.9%(15/43)  ※標準化必須(なしだと20.9%まで低下)
```

数式回帰(33.3%)と同水準に留まりNetLSD検索(50〜51%)には届かず。理由:
数式プロファイルは「その行自身の戦略ごとの違い」を直接エンコードした値なので、検索で
借りても数式回帰と同じ情報を再利用しているに過ぎず性能の天井も同じ。検索型が万能なの
ではなく、**検索キーが数式と独立な情報を持っているかどうか**が効いていた。

### 古典ML(RF/SVM/線形回帰)との比較
TabPFNの優位性が本物かを確認するため、同じ特徴量セットをRandomForest/SVM/線形回帰でも検証:

```
                    Linear   RF      SVM     (参考)TabPFN
数式5特徴量          37.5%   29.2%   16.7%    33.3%
NetLSD20次元のみ      4.2%    4.2%    4.2%     4.2%
数式+NetLSD(25次元)  37.5%   27.1%   12.5%    27.1%
```

単純な線形回帰がTabPFNをわずかに上回り(この規模・この特徴量数ではモデルの複雑さより
安定性が勝つ)、NetLSDのみだとモデルを何に変えても4.2%で完全に一致(モデル性能の差では
なく「グループ内分散ゼロ」という構造的問題だとモデル非依存に裏付けられた)。SVMは
ハイパラ未調整のため軒並み悪い。結論: モデル選びより特徴量/手法の選択の方が支配的。

### NetLSDと数式を「合体」させる3通りの試み→すべて失敗、NetLSD単体が最良のまま確定
1. **回帰の特徴量として合体**(前述、27.1%に悪化)
2. **検索キーとして40次元に合体**: 数式プロファイル単体と完全に同じ数字(34.9%)になった。
   NetLSDの20次元は同じ熱トレース曲線の近接時間サンプルで実質的な自由度が低く、
   z標準化後も数式プロファイル側(相関の低い20次元、実効自由度が高い)に距離計算が
   支配されてしまうため
3. **距離をalpha×NetLSD距離+(1-alpha)×数式距離でブレンド**(各距離は訓練データ内
   平均距離で正規化): alpha=0.1〜0.9の中間帯はすべて30〜35%に張り付き、alpha=1.0
   (NetLSD単体)にした瞬間だけ48.8%(k=3)/51.2%(k=5)に跳ね上がる、という非連続な
   挙動。数式距離は少しでも混ぜるとNetLSDの正しい近傍選択を狂わせるノイズとして
   働くと判明

3通りとも「合体は損」という一貫した結果になり、**NetLSD単体でのk-NN検索(k=5、
43グループ版で51.2%)が今日の最良解**として確定した。ただしワークロード6種
(BT/FT/IS/MG/CG/SP)・グループ43〜48個(leave-one-workload-outで実質6フォールド)
という小さい母数での結果であり、統計的にはまだ幅がある。`run_tonight.py`完了後の
8ワークロード×2クラスの新データで再検証する価値がある。

## 16. 重大な訂正: 多数派クラスチェックと、二段階分類による突破

### 「ランダム=20%」という比較基準そのものが間違っていた
15節までの評価は全て「5択ランダム=20%」を基準にしていたが、実際の正解ラベルの分布は
Scatterが45.8〜46.5%を占める偏った分布だった(BT/FT/IS/MG/CG/SP全体で
Scatter=22, MPO=10, HPO=8, Packed=6, EPO=2)。つまり**「常にScatterと予測する」
という何も学習していない一行のルールが、既に45.8%を叩き出す**。これを基準に
今日の全手法を再評価したところ:

```
多数派ベースライン(常にScatter):        45.8%
数式単体argmin:                        20.8%  (大敗)
数式回帰(TabPFN, 4特徴量、15節の旧ベスト): 33.3%  (敗北)
線形回帰:                              37.5%  (敗北)
RF:                                    27〜29%  (敗北)
SVM:                                   13〜17%  (敗北)
NetLSD検索(正規化後、修正版):           37〜44%  (敗北)
NetLSD検索(元の未正規化バグ版、旧報告51.2%): 実際は50.0%で辛うじて+4.2pt
  → ただし正解がScatter以外のグループでの的中率は19.2%(ランダム20%以下)
```

つまり**15節で「最良解」と呼んでいたNetLSD検索51.2%も含め、今日試した単純な
(1段階の)手法は実質的にどれも「多数派クラスへの逃げ」の域を出ておらず、
Scatter以外が正解の場面ではランダム同然だった**。これは15節の結論を大きく
訂正するものであり、正しい比較対象は「ランダム20%」ではなく「多数派45.8%」
だったと明記しておく。

### 突破口: 二段階分類(Scatterか否か→残り4択)
ユーザー発案の二段階分類(Stage1: Scatterか否かの二値判定、Stage2: 外れた場合に
残り4戦略=EPO/HPO/MPO/Packedの中で判定)をk-NN検索で試したところ、初めて
多数派ベースラインを実質的な意味で超えた:

```
                              全体     Stage1二値正解率  非Scatter的中率
正規化/正規化                  39.5%     69.8%            13.0%
未正規化/未正規化(volume込み)   48.8%     72.1%            34.8%  ← 最良
未正規化/正規化                37.2%     72.1%            13.0%
正規化/未正規化                48.8%     69.8%            30.4%

(参考) 多数派ベースライン: 46.5%
```

**Stage2(Scatter以外のどれかを絞り込む段階)では、未正規化(通信量の絶対値込み)
のNetLSD記述子が明確に優位**(34.8% vs 13.0%)。Stage1(Scatterか否かの二値)では
正規化・未正規化どちらでも大差なし(69.8% vs 72.1%)。

### 「正規化しないこと=悪」ではなかった
15節では未正規化ラプラシアン(通信量の絶対スケールが固有値に乗る)を「バグ」と
位置づけたが、二段階分類の結果から見直すと、これは間違いではなく**別の情報を
測っていただけ**だったとわかった。総通信量が多いほど帯域競合が起きやすく
Scatterが有利になる、というルーフラインのbandwidth_bound特徴量と同じ物理的
直感に合致する情報であり、特にStage2(Scatter以外の4戦略の絞り込み)では
むしろ捨ててはいけない情報だった。正規化(サイズ・順列不変性)とその放棄は
一方的な優劣ではなく、**どの段階でどちらの情報が必要かに応じて使い分けるべき**
という結論になった。

**評価**: 二段階分類(未正規化/未正規化)の48.8%は、多数派ベースライン(46.5%)を
+2.3pt上回るだけでなく、非Scatter的中率が34.8%(ランダム20%を大きく上回る)と、
今日初めて「Scatter以外を本当に区別できている」実質的な証拠が得られた。ただし
43グループ中1グループ程度の差であり、統計的にはまだ脆い。`run_tonight.py`完了後の
新データでの再現性確認が引き続き必要。

### 「行単位k-NN回帰」は不発、「次元数・k値の見直し」で53.5%まで改善
サンプル数不足(EPO=2件、Packed=4件)への対策として、258行(43グループ×5戦略)全部を
訓練データにする行単位k-NN回帰を試したが、グループ間(ワークロード・スレッド数による
桁違いの差)の分散がグループ内の戦略間分散を圧倒してしまい、グループ内正規化
(最速戦略を1.0とする比率に変換)を施しても改善せず、最良でも30.2%(数式5特徴量,k=8)
に留まった。「行を増やす」だけでは希少クラス問題は解決しないと判明。

一方、二段階分類のNetLSD次元数(3〜20)を再確認したところ、**未正規化版は次元数を
変えても結果が完全に不変**(volume情報が支配的で、8次元以降飽和するという15節の
知見の通り)。ただしその過程でStage1のk値をk1=5からk1=3に変えたところ、
**48.8%→53.5%まで改善**(非Scatter的中率も34.8%→43.5%)。次元数自体は無意味
だったが、k値の見直しという副産物で今日の最良解が更新された:

```
未正規化(volume込み)/未正規化、次元数3〜20(全て同一)、k1=3・k2=3:
  全体的中率 53.5%(多数派ベースライン46.5%を+6.7pt上回る)
  非Scatter的中率 43.5%
```

### 古典アルゴリズム(RF/SVM/線形)でも二段階分類を試す→k-NNに明確に劣る
NetLSD記述子(未正規化・正規化それぞれ)を特徴量として、Stage1(Scatterか否かの
二値分類)・Stage2(残り4択)をRandomForest/SVM/ロジスティック回帰でも実施し、
k-NN(53.5%)との比較を行った:

```
最良(RF, 未正規化/未正規化・両段):        37.2%(多数派ベースライン46.5%にも届かず)
SVM・ロジスティック回帰(全組み合わせ):     11.6〜18.6%(壊滅的)
(参考) k-NN二段階分類(未正規化/未正規化, k1=3,k2=3): 53.5%
```

原因: Stage2(非Scatterの4択)の訓練データは約17〜20件を4クラスに分けたものしかなく、
RF/SVM/ロジスティック回帰のような「全体に一つの決定境界を学習するモデル」はこの
極小データではほぼ機能しない。k-NN(検索型)は「一番似ている前例を1つ借りてくる」
だけなのでこの環境でも機能した。**「二段階に分解したこと」だけでなく「k-NN(検索型)
という手法自体」も、53.5%という結果を支える不可欠な要素だった**とわかった。これは
14節以降繰り返し確認されてきた「この規模のデータでは複雑なモデルより単純な仕組みが
勝つ」という教訓の、二段階分類版での再確認でもある。

**今日時点での最終結論**: 二段階k-NN検索(未正規化NetLSD、k1=3・k2=3)による**53.5%**
(多数派ベースライン46.5%比+6.7pt、非Scatter的中率43.5%)が最良解。ただし
ワークロード6種・43グループという小さい母数であり、`run_tonight.py`完了後の
8ワークロード×2クラスの新データでの再現性確認が必須。

## 17. 検証編: 過学習の発覚とNested CVによる決着

### 別バックアップ(.backup/sizeS + .backup/sizeW)での再現性検証→大きく崩壊
16節の53.5%(k1=3,k2=3)が本物かを確かめるため、`.backup/.backup`とは別の
`.backup/sizeS`・`.backup/sizeW`(BC/BFS/BT/CC/CG/FT/IS/MG/PR/SP/SSSP/TC/lavaMD、
2026年6月30日頃のスナップショット)でNPB系(BT/CG/FT/IS/MG/SP)のみに絞って
同じ手法を検証した。結果、同じ(k1=3,k2=3)は45グループ中20.0%まで崩壊し、
k1/k2を振り直しても46.7%が精一杯(多数派ベースライン51.1%にも届かず)だった。
命令数チェック(GAPBS型データ破損の有無)は健全だったが、このバックアップは
今週(NUMA構成確定・MPOの本物のDeLocアルゴリズム実装等)の変更より前の
スナップショットであり、条件が違いすぎる・過去のSVM実績(84.6%)とも矛盾する
再現性のなさから、**最終的にユーザー判断でこのバックアップは検証対象外**とした。

同じ時期に、2026年7月1日資料の28特徴量(実測ハードウェアカウンタ、Packed戦略
1回の実測から抽出)による過去のSVM実績(ALLTH 84.6%)も再検証したが、
二段階k-NN(37.8%)・二段階古典アルゴリズム(44.4%)いずれも当時の数字を再現できず、
これも「条件が違うデータでの過去の実績はそのまま信用できない」という教訓の
一例として記録に残す。

### 「70%」の正体: Stage1(二値判定)だけの精度であり、最終正解率ではない
ユーザーから「43グループで70%くらい出ていなかったか」との指摘があり確認したところ、
これはStage1(Scatterか否かの二値判定)単体の正解率(69.8%〜74.4%)であり、
最終的な5択の的中率(53.5%〜55.8%)とは別の内訳指標だったと判明。二値判定自体は
比較的高精度だが、そこから最終的な戦略決定に落とし込む過程(Stage2)で精度が
落ちる、という二段階構成の内訳が改めて明確になった。

### ピックアップ手法の掘り下げ(2択・3択・4択)
ユーザー発案で、Stage2の投票を「ペア(2択)」「トリオ(3択)」「カルテット(4択)」の
ミニトーナメント集計に変えて比較した:

```
ピックアップ2(ペア): 非Scatter的中率 13.0%(大きく悪化)
ピックアップ3(トリオ): 非Scatter的中率 43.5%(回復)
ピックアップ4(カルテット=単純な1位ラベルだが訓練プール無制限): 非Scatter的中率 43.5%(同着)
```

ピックアップ3と4が完全に同着だったことから、**改善の正体は「ペア→トリオ」という
分解の精緻化ではなく、「Stage2の訓練プールをScatterが1位のグループも含めた
全グループに広げたこと(以前は非Scatter勝ちグループだけに絞っていた)」**だったと判明。
オッカムの剃刀的に、一番単純な構成(ピックアップ4)で十分だった。

### Nested CV(入れ子交差検証)による誠実な再評価→51.2%に着地
今日発覚した過学習パターン(k1・k2を報告用スコアに直接総当たりして選んでいた)を
是正するため、外側6フォールド(leave-one-workload-out)×内側5フォールドの
入れ子交差検証を実装。外側テストワークロードのデータを一切使わずに内側CVで
k1・k2を選び、それを外側テストに適用する、という誠実な手順に直した。

```
直接チューニングした最良値(過学習込み): 55.8%
Nested CVによる誠実な最終的中率:        51.2%(22/43)
多数派ベースライン:                    46.5%
```

過大評価分はおよそ半分(+9.3pt→+4.7pt)だったが、**多数派ベースラインを上回る
本物のシグナルは残った**。以後、43グループデータでの評価はこのNested CV
方式を標準とする。

### 別のグラフ類似度(手作りグラフ統計量)との比較→NetLSDが引き続き優位
NetLSD(固有値スペクトルの熱トレース)以外の切り口として、総通信量・ノード強度の
ばらつき(変動係数)・重み付きクラスタ係数・最大/平均次数比・エッジ重みのGini係数、
という6個の手作りグラフ統計量を同じNested CVで検証したところ37.2%
(多数派ベースライン46.5%にも届かず)。NetLSDが「たまたま今日のデータに
合っていた」のではなく、スペクトル分解という切り口自体に意味があることが
裏付けられた。

### 数式×NetLSDの確率的合体(ベイズ的事後確率)→4回目の合体失敗
特徴量結合・検索キー結合・距離ブレンドに続き、数式(順位ベースの事前確率)と
NetLSD(近傍k個の投票=尤度、ラプラススムージング)を`P_final ∝ P_formula × P_netlsd`
という確率の掛け算で合体させたが、Nested CVで39.5%となり、NetLSD単体
(単一段階5択、46.5%)にも及ばなかった。特徴量・距離・確率という3つの異なる
数学的枠組み全てで合体が悪化するという一貫した結果であり、数式とNetLSDは
補完的に働かないと言ってよい。

また、この検証の副産物として、**単一段階の5択NetLSD投票(46.5%)は二段階分類版
(51.2%)に届かない**ことも判明。今日発見した中で本質的に効いていたのは
「グラフ類似度の種類」でも「数式との合体方法」でもなく、**「Scatterか否かを
まず分けてから残り4択を考える」という二段階構成そのもの**だった。

### 最終結論(Nested CV基準、43グループ、NPB系のみ)
```
1位: NetLSD(未正規化) + 二段階分類 + Nested CV        51.2%
2位: 多数派ベースライン / NetLSD単一段階(5択)          46.5%
3位: ベイズ合体(数式×NetLSD)                          39.5%
4位: グラフ統計量 + 二段階                             37.2%
5位: 数式単体(順位ベース)                              20.9%
```
`run_tonight.py`完了後、8ワークロード×2クラスの新データでこの順位が
再現するかの検証が今後の課題として残る。
