## 1. sizeA「重い組」(BT/MG/SP) が2THで進まない問題の再調査

### 発端
- 7/3に分割した「軽い組」は全ワークロード完了済みなのに、「重い組」(`--threads 2 6 8 12 --workloads BT MG SP --no-timeout`, PID 1398035) は2THのままMGの4戦略のみ完了（各4〜4.6時間）し、BT/SPの8戦略が21.5時間経過しても終わらない状態だった

### 誤った初期仮説（訂正済み）
- 「`get_cpu_map()`がワークロードを見ずに戦略名だけでコア番号を返すため、BT/SPが同じ戦略なら同じ物理コアに重複ピン留めされ、host core 0番に実質6プロセスが密集している」と推測
- しかし `taskset -cp <pid>` で実際に確認したところ、21.5時間CPUを食っている `/root/sniper/lib/sniper` 本体プロセスは**全て affinity list = 0-23（ピン留めなし）**だった
- `GOMP_CPU_AFFINITY` はGNU OpenMPランタイム向けの環境変数で、効くのは「ベンチマークバイナリを実機で走らせてPinで命令トレースを取る」フェーズ（`record-trace`/`pinbin`、CPU時間0:00で瞬時に終わっている）のみ。実際に何時間も走る`sniper`本体（トレースリプレイによる詳細タイミングシミュレーション）はこの環境変数の影響を受けず、自由にスケジューリングされる
- → **物理コアの奪い合いは起きていなかった**。理論を撤回

### 本当の原因
- `run_profile.json` に **BT_A・SP_A のデータが一度も存在しない**（今回が初計測）
- BT/SPはNPBの中でも class A での演算量増加が特に大きく、かつ2スレッドは1スレッドあたりの担当命令数が最大になる条件（＝最も遅い条件）
- `orchestrator.py` の `main()` はスレッド数ごとに**完全に逐次実行**するため、`--threads 2 6 8 12` の並びだと最も遅い2THが終わるまで6/8/12THに一切進めない、という設計上の問題が主因
- size Wの実測データ（BT_W: 2TH=6175s→12TH=2283s、SP_W: 2TH=14094s→12TH=5806s、MG_W: 2TH=1545s→12TH=572s）でも、2THが一貫して最も遅く12TH付近が最速という傾向を確認。ただし6TH→8TH、12TH→16THの間は非単調（ホスト混雑などのノイズ）

### 対応
- Outputs/sizeA/2TH内の**7/2分の古いディレクトリ44件を削除**（BC/BFS/CC/CG/FT/IS/MG/PR/SSSP/TC/lavaMDの4戦略×11workload。BT/SPは7/2分自体が存在しなかった）
- 既存の2TH実行（PID 1398035）はそのまま継続
- 新規に `--threads 12 8 6 --bench-class A --workloads BT MG SP --no-timeout` を裏で起動（PID 3383558、ログ: `logs/heavy_highTH.log`）。12THから降順に着手することで、先に速く終わる高スレッド数のデータを確保する狙い

```bash
nohup python3 orchestrator.py --threads 12 8 6 --bench-class A \
  --workloads BT MG SP --no-timeout > logs/heavy_highTH.log 2>&1 &
```

## 3. スケジューリング理論の導入検討

### 問題の定式化
- SizeBの推定計算時間が出揃った後、2/6/8/12THのジョブ群を「投入する順序・詰め方」で全体完了時間を最短化したい、という相談
- これは資源制約付きスケジューリング（Garey & Graham, 1975, "Bounds for Multiprocessor Scheduling with Resource Constraints"、$P\,|\,res\,1\,|\,C_{\max}$）そのもの。NP困難だが、
  - **List Scheduling + LPT（長い順に、空いた瞬間に詰める）**で最適解に対し $(2-1/C)$ 以内の近似保証
  - ジョブ数が少数（数十件）なら OR-Tools CP-SAT の `cumulative` 制約で厳密最適も現実的に解ける
- 実装はSizeBのBT等が終わり推定値が揃ってから着手する方針（今回は理論と効果測定のみ）

### SizeWの実測データで「現行の二重ループ」vs「厳密スケジューリング」を比較
`run_profile.json` のsize Wデータ（322ジョブ = 13workload×4戦略×6スレッド数）を使い、makespanをシミュレーション。

| 資源容量 | 単純二重ループ（現行） | 制約付き厳密（LPT） | 短縮 |
|---|---|---|---|
| C=32（コードの想定値） | 28.15h | 22.49h | 20.1%（5.66h減） |
| C=24（実機コア数） | 38.97h | 32.38h | 16.9%（6.59h減） |

- 単純二重ループの最大のボトルネックは、C=24時に`16TH`フェーズが`workers=24//16=1`となり**54ジョブが完全直列**になる点（63,745s=17.7時間）。スレッド数の境界で並列化の余地を捨てているのが無駄の正体

## 4. Sniper本体のホストCPU消費はスレッド数に比例しないことが判明

### 発端
- `_HOST_CORES=32`だった理由は「単純な二重ループだとスレッド数境界で遊休コアが出やすいので、あえて実機24コアを超える32で見積もることで全コアを稼働させる工夫だった」という開発経緯が判明
- 厳密スケジューリング導入後は容量をどう設定すべきか（24に戻す？ SizeB計測中の機械学習用にもっと減らす？）を検討

### 実測
`ps -eo pid,pcpu` で実行中プロセスのCPU%を確認：

```
2TH ジョブ（BT/SP, PID 1398869等）:  CPU使用率 平均121%  → 実コア消費 ≈ 1.2コア
12THジョブ（BT/MG, PID 3383680/3383702）: CPU使用率 平均327% → 実コア消費 ≈ 3.3コア
```

- スレッド数は6倍（2→12）なのに、実ホストCPU消費は2.75倍（1.2→3.3コア）にしかなっていない
- Sniperの詳細タイミングシミュレーションはターゲットのスレッド数に対して**準線形**にしかホストCPUを使わない
- **重要な誤解の訂正**: この数値は「ワークロードが何倍速くなったか」（研究上の実験結果、size Wで確認済みの2.7倍速など）とは別軸。Sniper本体という計算エンジン自身がホストCPUをどれだけ使うかという、全く独立した指標

### べき乗フィットによる換算テーブル（推定・要検証）
実測2点（2TH, 12TH）から `cost(threads) = a × threads^b` をフィット：

```
cost(threads) ≈ 0.811 × threads^0.565
```

| スレッド数 | 実消費コア換算 | 備考 |
|---|---|---|
| 2TH | 1.20 | 実測 |
| 4TH | 1.77 | 推定 |
| 6TH | 2.23 | 推定 |
| 8TH | 2.62 | 推定 |
| 12TH | 3.30 | 実測 |
| 16TH | 3.88 | 推定 |

※4/6/8/16THは2点だけからの外挿。SizeAの新バッチ（12→8→6TH）が進んだ際に実測して検証・補正する必要あり。

### 実消費コア換算での再計算（size W, 練習）

| モデル | 資源容量 | 合計時間 | 現行比 |
|---|---|---|---|
| 現行の二重ループ（width=スレッド数） | C=32 | 28.15h | — |
| 厳密スケジューリング（width=スレッド数のまま） | C=24 | 32.38h | 悪化 |
| 厳密スケジューリング（実消費コア換算） | C=24 | **9.31h** | **-66.9%** |
| 厳密スケジューリング（実消費コア換算・ML学習用に4コア確保） | C=20 | **11.26h** | **-60.0%** |

- width=スレッド数のまま容量だけ24に減らすと、16THジョブが幅16を占有してしまい詰め込みがほぼできず**むしろ悪化**する
- 実消費コア換算（16TH≈3.88）を使うと、同じ24コアに理論上6本近く同時に詰められるようになり、28時間規模の計算が9.3時間まで縮む計算結果になった

### 結論・今後の方針
- `_HOST_CORES=32`という水増しは、厳密スケジューリング導入後は不要になる（遊休対策という当初の目的自体が厳密スケジューラで解消されるため）
- ただし width をスレッド数のまま容量だけ実機値(24)に戻すのは不正確（低スレッド数ジョブの実コスト過大評価、高スレッド数ジョブの過小評価が起きる）
- **正しい設計方針**: widthは実測CPU%換算値を使い、容量は実機コア数（24）を基準に、SizeB計測中の機械学習学習ジョブ用に2〜4コア分を差し引いた**実効容量20〜22**を使うのが妥当
- 4/6/8/16THの換算値は推定のみなので、実測が取れ次第フィット式を補正する

## 6. ワークロード別の実消費コアを実測（SizeSで全13ワークロード）

SizeAの2バッチを止めずに、SizeSで軽く1本ずつ回してCPU%を実測。

### 方法論上の重要な落とし穴
`ps`の`%CPU`は**プロセス生存時間全体の累積平均**であり瞬間値ではない。起動直後の低使用率区間（コンテナ起動・Pinトレース記録）が平均に混ざるため、短時間で終わるジョブほど過小評価される。実際、MG_S_2の初回計測で「19秒のジョブなのに%CPUが7〜17%」という値を掴み、一瞬「ハングした」と誤診した。高頻度ポーリング（0.2〜0.5秒間隔）でやり直したところ、他のNPB系と同様に0%から単調に収束していく途中だっただけと判明。

### 実測結果（SizeS・2TH）
| ワークロード | 実消費コア(%) | 備考 |
|---|---|---|
| BT | 133 | 収束確認済み |
| FT | 117 | 収束確認済み |
| SP | 116 | 収束確認済み |
| IS | 102 | 収束近似 |
| MG | 100 | 収束近似（高頻度ポーリングで確認） |
| CG | 89 | 収束確認済み |
| BFS/PR/BC/CC/SSSP/TC | 57 | GAPグラフ系、構造がほぼ同一なのでBFSを代表測定 |
| lavaMD | 13 | 測定困難なほど実行時間が短い(4〜5秒) |

**発見**: NPB系（89〜133%）とGAPグラフ系（57%）で実消費コアに明確な差がある。widthはスレッド数だけでなくワークロードの種類にも依存する。

## 8. SizeB全ワークロード×全スレッド数の厳密スケジューリングを試算

6〜7章の実消費コアモデルを使い、OR-Tools CP-SAT（`cumulative`制約、容量20コア=ML学習用に4コア確保）でSizeB全体の最短完了時間を試算。

### 重要な前提（要注意）
BT/CG/FT/IS/MG/SPの6ワークロード（NPB系）は、現行の`estimate_walltime()`がclass Bへの実データを持たないため、**全スレッド数で一律14400秒（4時間キャップ）** に張り付く（78組み合わせ中36組み合わせ、うち144ジョブがこの巨大ブロック）。GAPグラフ系とlavaMDは実データからの妥当な推定値（141〜215秒、11〜13秒）が出ている。

### 試算結果
| | 時間 |
|---|---|
| 全312ジョブを直列実行した場合 | 585.5h（24.4日） |
| ベースライン（現行コード相当：スレッド数ごと完全逐次、width=スレッド数、C=32） | 163.0h（6.8日） |
| 厳密スケジューリング（width=実消費コア換算、容量20、CP-SAT） | **56.0h（2.3日）** |
| 短縮率 | **約65.6%** |

CP-SATは45秒（2ワーカーに制限してSizeAへの負荷を抑制）でFEASIBLE解を発見。理論下限（総コア秒÷容量20＝53.6h）との差はわずか4.5%で、ほぼ最適に近い。

### 結論
- 「スレッド数ごとの完全逐次実行」をやめて「実消費コアベースで容量いっぱいに詰め込む」だけで、**約65%の時間短縮**という手法自体の効果は確認できた
- ただし絶対値の56時間はNPB6種が軒並み4時間キャップに張り付いた状態での計算であり、プレースホルダーに近い。BT_A/SP_Aが完了しSizeBへの本当の倍率が判明すれば、この数字は変わる（おそらくもっと長くなる）
- SizeAが完了し次第、本試算をリファレンスデータで差し替えて本格的な推定を行う方針

## 9. vsPOSM: POSMとの公平な比較、およびMPOの正体調査

### 発端
`MachineLearning/vsPOSM`ディレクトリでPOSM(先行研究)とTabPFNを比較する作業を開始。POSM本来の出力はMPO/HPOの2択（`Documents/jinyifan_master_thesis.pdf`確認済み）であり、既存の`posm_baseline.py`が4クラス正解に対してPOSMを評価していたのは構造的に不公平だった問題（詳細は[[project_vsposm]]参照）。

### MPO自動複製ロジックをorchestrator.pyに実装
`utility/cpu_affinity.py`の`MPO_MAPS`と`STRATEGIES`を突き合わせた結果、CG/BT/IS/SP/EPはHPOと、FT/BFS/PR/BC/CC/SSSP/TC/lavaMDはScatterと、cpu_mapが完全一致（MGだけ独自パターンで4/6/8/12/16THは不一致）と判明。これを受けて:
- `resolve_mpo_equivalent(workload, num_threads)`を`cpu_affinity.py`に追加（cpu_map一致判定）
- `orchestrator.py`にMPO自動複製ロジックを実装（一致すれば実シミュレーション省略、複製して`PRESET=MPO`等を書き換え）。今後`--strategies`に`MPO`を含めても無駄な実行が起きない
- SizeS/SizeWの146ディレクトリをコピー・リネームで用意、MGだけ実測（6,8,12THはSizeS/SizeW両方で本実行、2THはPacked流用）
- 4TH/16THと`MachineLearning/{TabPFN,RandomForest,SVM,XGBoost}/4TH,16TH`は現行実験・学習の対象外と判明し`.backup/`に移動

### 重大な発見: MPO/HPOの実測差がほぼ無い
`vs_posm.py`でSizeS/SizeWの2クラス(MPO/HPO)データを集めたところ、**6TH/8THでは全ワークロードの差が1%未満**（ノイズレベル）、2THのみやや差がある程度だった。TabPFNを回す前にこの異常に気づき停止。

### さらに重大な発見: 論文の「Jinの2倍差」はHPO vs Round-Robinだった、そしてMPOの前提が食い違う
- ユーザー指摘の「lavaMDでMPO/HPOに2倍近い差」を確認するため論文のFigure 4.13（p.46）を画像で直接確認 → **本物のMPO対HPO直接比較**で、lavaMD(MPO=0.97, HPO=0.51)・NPB-BT(MPO=0.90, HPO=0.56)に実際大きな差があることを確認
- しかしこれは**うちの`MPO_MAPS`の前提と矛盾**する。BTのMPOはコード上HPOと完全一致するcpu_mapとして定義されているが、論文ではBTの実測通信行列(Figure 4.5)に基づき(0,1)(2,3)(4,5)ペアが突出して通信量が多いことが分かり、これをペアリングする「本物のMPO」はHPOとは別物のはず
- lavaMDも同様、コードでは"MPO=Scatter"と仮定しているが、論文のグラフではMPO(0.97)はPacked(1.0)にほぼ等しく、Scatterとは違う挙動

### Jinの本当のMPOアルゴリズム（論文3.3.1節で確認）
1. タスク間の**実測通信量**でペアを作る（通信が多い2タスクを同ノードへ）
2. ペアを、同時に発生する通信が別ノードになるようグループ分け（輻輳回避）
3. ノード内では負荷の大きいタスクをBig(P)core、小さいタスクをSmall(E)coreに配分

これはデータ駆動（実プロファイリングで得た通信行列が必要）。うちの`MPO_MAPS`はワークロードのアルゴリズム的性質からの**静的な推測**であり、根本的に別物と判明。

### 通信行列の出典を特定: DeLoc論文とdetloc-tracerツール
Jinの論文自身には通信行列の具体的な計測手法が書かれておらず、参照[1] **Agung et al., "DeLoc: A locality and memory-congestion-aware task mapping method for modern numa systems", IEEE Access 8 (2020)** から手法を借用していると判明。さらに調査したところ、東北大学高瀬研のGitHub `Tohoku-University-Takizawa-Lab/detloc-tracer` に実装（Pin toolベース）が公開されていることを発見:
- 全メモリRead/WriteをPinでフック（`INS_IsMemoryRead`/`INS_IsMemoryWrite`）
- アドレスを64byte単位のキャッシュライン番号に変換(`line = addr >> 6`)
- 各ラインの直前アクセススレッドを記録し、別スレッドが後からアクセスしたら「通信」としてカウント（producer/consumer方式の共有検出）
- スレッドペア×ペアの通信行列をCSV出力

うちの環境はPin 3.11、detloc-tracerの動作確認済みは3.5〜3.7とバージョン差があり、ビルド可否は未確認。

## 10. detloc-tracerのビルド・実行に挑戦 → Pinとホストカーネルの根本的な非互換性を特定

`/home/hiragahama/ClaudeXSniper/.outside_program/DeLoc` に`detloc-tracer`(GitHub: `Tohoku-University-Takizawa-Lab/detloc-tracer`)をユーザー承認の上でクローンし、実際にビルド・動作確認を行った。

### ビルド（成功）
Sniper付属のPin 3.11でビルドを試みたところ2つの問題があり、修正して成功:
1. `using namespace std;`が無く`cout`/`endl`関連のコンパイルエラー → 追加して解決
2. `Makefile`が`/usr/lib64/libelf.a`（静的ライブラリ）を要求するが、CentOS6コンテナの`elfutils-libelf-devel`は動的ライブラリ(`.so`)しか提供しない → 一度動的リンクに変更したところ、今度は**Pin自体が独自のサンドボックス化されたランタイム(PinCRT)で動くため実行時に`dlopen`できない**ことが判明。`elfutils-libelf-devel-static`パッケージ（`.a`を提供）をインストールし、元通り静的リンクに戻して解決

### 実行時の問題: マルチスレッド検出ができない
lavaMD（`-cores 2`）に対して実行したところ、`NUM_THREADS: 1`しか検出されず、通信データが空になる問題が発生。切り分けのため以下を実施:

| テスト対象 | 結果 |
|---|---|
| 自作の単純なpthreadプログラム | ✅ 2スレッド正常検出 |
| 単純なOpenMPプログラム(GOMP) | ✅ 2スレッド正常検出 |
| OpenMP + sched_setaffinity | ✅ 2スレッド正常検出 |
| OpenMP二重並列領域(短い領域→長い領域) | ✅ 2スレッド正常検出 |
| OpenMP + private()節 | ✅ 2スレッド正常検出 |
| OpenMP + libm(exp())呼び出し | ✅ 2スレッド正常検出 |
| Pin公式サンプルツール(`malloc_mt`)でlavaMD | ❌ 1スレッドのみ |
| 本物のlavaMD（プリビルド版・自前リビルド版とも） | ❌ 1スレッドのみ |

### 根本原因の特定: Pin本体のクラッシュ
`ulimit -c unlimited`と`coredumpctl`（ホスト側のsystemd-coredump）でクラッシュ時のコアダンプを解析した結果、**Pin本体(`pinbin`)内部でSIGSEGV**していることが判明:
```
Program terminated with signal SIGSEGV
#0  0x00007f3a65c422b6 in ?? ()  (/opt/pin37/intel64/bin/pinbin + 0x34f2b6)
```
私たちのツールコードでもlavaMDのコードでもなく、Pin自身の内部インジェクション機構がクラッシュしている。lavaMDの2本目スレッド(GOMPワーカー)がこのクラッシュを引き起こす特定のコードパスを踏んでいる。

### バージョン変更では解決しない
Pin 3.11 → Pin 3.7（Intel公式サイトから取得） → Pin 3.5（同）と3世代試したが、**すべて同一箇所で同一のクラッシュ**。さらに調査したところ、Pin 2.14系は公式に「カーネル4.x以降非対応」と明記されていることが分かった。うちのホストはLinux 6.19という非常に新しいカーネルであり、Pin 2.14(2015年頃)〜3.11(2019年)まで**Pinのバージョンを問わずこのホストカーネルとは根本的に非互換**という結論に至った。

### 結論
- podmanなどのコンテナはホストとカーネルを共有するため、コンテナ内のOSイメージを変えてもカーネルバージョンは変わらない（VMとの違い）。今回のPin非互換性はまさにこのカーネル層の問題
- 本当に動かすにはQEMU/KVM等の本物のVMで当時想定されていた世代のカーネルを用意する必要がある（`/dev/kvm`は利用可能と確認済みだが、大掛かりな作業になるため今回は見送り）
- **detloc-tracerによる本物のMPO通信行列の取得は、現環境では実現不可能**と判断。[[project_vsposm]]のMPO_MAPSは「うちの独自解釈」として扱い、論文の本物のMPOとは別物であることを明記して進める方針とする

## 14. `utility/deloc_mapper.py` 実装：Jinの本物のMPOを再現

### 当初の実装とユーザーからの指摘
`spatial_deloc.py`のDeLoc本体（ノード決定のみ、コア種別は無関係）を本プロジェクトのトポロジ（2node×4P+4E）に移植して`deloc_mapper.py`を作成。動作確認のためlavaMDの実データに適用したところ、SizeA・12THだけノード配分が5-7に崩れる現象を発見（通信負荷が偏った結果、物理コア上限より先にロード基準の分岐が効いた、アルゴリズム通りの挙動）。

この過程で、Jinの修士論文Figure 4.13（lavaMD: MPO≈Default、HPO≈半分）とうちの実データ（DeLoc計算結果はScatter/HPO相当で高速）が食い違う点をユーザーが指摘。調査の結果:
- トポロジは論文本文（4.2節）とうちの`cpu_affinity.py`が完全一致（0-3,8-11=Big/P、4-7,12-15=Small/E、2 NUMAノード）
- ただしP/E周波数比が違う：論文は2.4GHz/1.2GHz(2:1)、うちは4.0GHz/1.0GHz(4:1)
- 決定打は**論文3.3.1節本文**：JinのMPOはAgungの元のDeLoc（ノード決定のみ）に、**「ノード内で負荷の大きいタスクをBig core、小さいタスクをSmall coreへ割当てる」という第3ステップを独自に追加した拡張**だった。論文自身が「ノードのBig core数を超えた分は強制的にSmall coreに落ちる」という構造的欠陷を明記しており、これがMPOがHPOに劣る根本原因

### mem_access.csvの発見（負荷指標）
DeLocの通信行列には各スレッド自身の計算負荷が現れないが、detloc-tracerは同じ実行から`mem_access.csv`（tid, 読込数, 書込数, 読込バイト数, 書込バイト数）も出力していることが判明（`detloc_tracer.cpp`のソース確認）。既存の113ファイル全てで揃っており再取得不要。

### 最終実装：Step1(ノード決定)+Step2(ノード内Big/Small)
ユーザーからの「同種コア内の割当を考慮するのはズルいのでは」という指摘を受け、一度は既定を「負荷を見ないフェアなMPO」に変更したが、論文本文の確認によりJinの本物のMPOが実際に負荷を考慮する仕様だと判明したため、最終的に以下の2ステップ構成に確定:

- **Step1**: 通信量のみでノード決定（Agungの元のDeLoc, weight_type=0）
- **Step2**: Step1のノード所属を変えず、ノード内で負荷降順にBig(P)→Small(E)へ再配置。ノードのBig core数を超えた分は強制的にSmall coreへ（論文の失敗例を忠実に再現）

`--node-only`フラグでStep1のみ（Agung単体のDeLoc）も選択可能。ノード制約自体を緩和する（他ノードの空きBig coreを借りる）改良はJinの原著にはなく、**意図的に未実装のままファイルに明記**（ユーザーの論文の新規性候補として温存。仮称: OMPO/EMPO等）。

## 15. サイズ依存性の検証：NPB・GAPBS・lavaMDで正反対の傾向

DeLocが計算するノード配分・負荷不均衡度（`imbalance_ratio = max/mean - 1`）が問題サイズ（S/W/A）でどう変わるかを、既存データのみで検証。

**NPB（BT/CG/FT/IS/MG/SP）: サイズが大きいほど不均衡が縮小**
| | SizeS | SizeW | SizeA |
|---|---|---|---|
| BT 8TH | 37% | 14% | 3% |
| SP 8TH | 37% | 21% | 4% |
| MG 8TH | 17% | 2% | 0% |

構造化された領域分割（格子分割）では、問題サイズが大きくなるほど境界(エッジ)処理の相対的な割合が縮小し、スレッド間の作業量が均されるため。

**GAPBS（BFS/PR/BC/CC/SSSP/TC）: サイズが大きいほど不均衡が激増（NPBと正反対）**
| | SizeS | SizeW | SizeA |
|---|---|---|---|
| BFS 12TH | 179% | 311% | 443% |
| BC 12TH | 142% | 283% | 413% |
| CC 12TH | 168% | 274% | 314% |

GAPBSのグラフはKronecker/RMAT型のべき乗則次数分布を持つため、頂点数が増えるほど一部の超高次数ハブ頂点を担当するスレッドだけが極端に重くなり、サイズを大きくしても均されるどころか悪化する。

**lavaMD: NPB寄り（サイズが大きいほど不均衡縮小傾向）**
| TH | SizeS | SizeW | SizeA |
|---|---|---|---|
| 6TH | 58% | 36% | 14% |
| 8TH | 49% | 22% | 30% |

粒子シミュレーションの箱分割はNPBの格子分割に近い構造のため同様の傾向。

**結論**: 「問題サイズに依存しない」という前提は成り立たず、依存の**方向性自体がワークロードのアクセスパターン（構造化された領域分割 vs 不規則なべき乗則グラフ）で正反対**になる。MPO_MAPSはサイズ別に計算し直す必要があり、この「構造化/不規則」という分類自体が負荷不均衡の予測因子になりうる新規性のある切り口。

## 16. 通信構造(comm.csv)と負荷不均衡(mem_access.csv)は別軸、それぞれサイズでどう変わるか

### JinのMPOに必要なデータの整理
Jin修士論文3.3.1節のMPOは2ステップで、使うデータが完全に分離している:

| データ | 使われる場所 | 答える問い |
|---|---|---|
| `comm.csv`(通信行列の疎密構造) | Step1(ノード決定) | どのスレッドを同じノードに置くべきか |
| `mem_access.csv`(負荷不均衡) | Step2(ノード内Big/Small割当) | ノード内でどのスレッドをBig/Smallに置くべきか |

どちらか一方では完全なcpu_mapは計算できず、両方が別々の役割で必須。

### 通信構造(comm.csv)自体のサイズ依存性
疎ペア(全体通信量の0.1%未満)の割合で見ると、**疎密の「量」自体はほとんどのワークロードでサイズにあまり依存しない**。BT/MGはSizeSでのみやや疎(7%/29%)だがW/Aでは0%に収束。CG/FT/IS/SP、GAPBS全6種は**全サイズで疎ペアほぼ0%**（8スレッドもあれば元から全ペアが密）。

唯一の例外が**lavaMD**で、疎ペア割合が36%→36%→46%と**全サイズを通じて疎な構造を保持し、むしろサイズが大きくなるほど疎さが増す**。箱分割による空間的局所性（近傍の箱としか相互作用しない）がそのまま表れている。

もう一つの一貫した構造として、**ほぼ全ワークロード・全サイズでthread0が通信の「ハブ」として最重量ペアに登場する**（BT/FT/IS/SP/GAPBS全6種）。この役割自体はサイズによらないが、thread0の「相手」（2番目に重いパートナー）はサイズごとに変わる。

### 負荷不均衡(mem_access.csv)のサイズ依存性: 「同じコアが悪化」か「違うコアが入れ替わる」か
最重量スレッドのID自体がサイズで固定か変動かを検証。**ワークロードの種類で全く違う答えになった**:

**GAPBS(悪化系): 一貫して同じスレッド(thread0)が加速度的に重くなり続ける**
| | SizeS | SizeW | SizeA |
|---|---|---|---|
| BFS 8TH | thread0(112%) | thread0(217%) | thread0(308%) |
| BC 8TH | thread0(98%) | thread0(184%) | thread0(279%) |
| CC 12TH | thread0(168%) | thread0(274%) | thread0(314%) |
| SSSP 12TH | thread0(132%) | thread0(237%) | thread0(334%) |

全ワークロード・全スレッド数で例外なくthread0が最重量。べき乗則次数分布のハブ頂点が一貫してthread0の担当領域に配置される実装のため、サイズが大きくなるほどそのハブ自体の次数が不均衡に伸び、thread0だけが際限なく重くなる。

**NPB(改善系): 最重量スレッドの識別自体がサイズごとに入れ替わる**
| | SizeS | SizeW | SizeA |
|---|---|---|---|
| BT 8TH | thread0(37%) | thread1(14%) | thread5(3%) |
| MG 8TH | thread7(17%) | thread0(2%) | thread6(0%) |
| SP 8TH | thread0(37%) | thread1(21%) | thread1(4%) |

最重量スレッドがバラバラに入れ替わるが、不均衡度自体が急速にゼロへ収束するため「誰が一番重いか」という問い自体がサイズが大きくなるほど意味を失う。

**lavaMDは中間的**(8THはthread3→3→2とやや安定、12THはthread5→3→3)。最軽量側(thread7/thread11)はむしろ一貫している。

### 設計上の含意
GAPBSは「特定の1コアが際限なく重くなる」構造的な偏りのため、DeLoc/MPOのStep2にとっては「常にthread0をBig coreに固定すればよい」という単純な対策が効きそうである。一方NPBは「誰が重いかはサイズ次第でランダムに近い」ため、事前の固定ルールが効かず、都度計算し直す必要がある。この違いは論文の新規性としても使える。
