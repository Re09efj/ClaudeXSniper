# 2026年7月13日

## 1. 発端: 教授からGCCの古さを指摘される

先生に相談したところ、「gem5と比較してSniper側のGCCが古すぎて、最適化が現代的
でないから論文に出せないのでは」との指摘を受けた。これを受けて、現行9種ワーク
ロード(BT/FT/IS/MG/CG/LU/canneal/dedup/GUPS)のビルド環境を全面調査した。

## 2. 現状調査: ビルド環境の実態

- `.bind`(Sniper開発用ソースツリー、`shell/sniper_dev_shell.sh`でbind mount運用)
  内のPin実行環境を確認したところ、コンテナ(`localhost/snipersim/snipersim:*`系
  podmanイメージ)はCentOS 6.10、GCCは`devtoolset-7`(7.3.1)。
- バイナリの`.comment`セクションを`readelf -p .comment`で確認した結果:
  - LU(旧NPB3.3-OMP): **gfortran 4.8.2**(2014年)でビルド
  - canneal/dedup/GUPS: **GCC 7.3.1**(2018年)でビルド
- 一方gem5側(`/home/hiragahama/gem5/ClaudeX/binary/NPB3.4.4/NPB3.4-OMP`)は
  **NPB3.4**(Sniper側はNPB3.3で1世代古い)を、ホストの**GCC 14.2.1/15.2.1**
  (静的リンク)でビルドしていることが判明。つまり先生の指摘は的中しており、
  「ベンチマークバージョンもコンパイラ世代も別物を比較していた」ことが根本原因
  だったと確定。

## 3. GCC15での静的リビルド(`binary/GCC15/`新設)

ホストにはFortran用のGCC14パッケージ(gfortran)が存在しない(dnf提供は
gfortran 15.2.1のみ、C/C++用のgcc14/g++14は別途存在)ことが判明したため、
ユーザー判断で**全ワークロードGCC15.2.1に統一**することに決定。ディレクトリも
当初作成した`binary/GCC14`から`binary/GCC15`にリネーム。

- **NAS系(BT/CG/FT/IS/MG/LU)**: 当初は現行Sniperと同じNPB3.3系列を保つため
  NASA公式(nas.nasa.gov)からNPB3.3.1を新規取得してビルドしたが、後に
  「どうせGCC15でリコンパイルするなら」という判断で**gem5側と同じNPB3.4ソース**
  に切り替えて再ビルド。`-fallow-invalid-boz`はNPB3.3特有の古い16進定数記法
  (`X'...'`)対応で必要だったが、NPB3.4では不要だった。
  最終的にS/A/Wの3クラス×6ワークロード=18バイナリ、全てVerification
  Successfulを確認。
- **canneal/dedup(PARSEC)**: `libstdc++-static`・`zlib-static`(Fedoraでは
  `zlib-ng-compat-static`という名前だった)をユーザーがsudo dnf installで導入。
  dedupは同梱openssl(`sha_locl.h`が`<openssl/sha.h>`を山括弧includeしており、
  システムのopenssl-develヘッダと型衝突する)問題があり、`-I.`でバンドル版を
  優先させて解決。
- **GUPS**: `-O3 -std=gnu99 -fopenmp -static`で静的ビルド。既存Makefileには
  「SIDのSniperコンテナにlibc.a等が無く-staticでリンクできない」という理由で
  `-static`を外すコメントが残っていたが、host側にglibc-staticが入っていたため
  今回は問題なくビルドできた。
- 全ワークロード`-O3`ベースで統一(旧GCC版も元々`-O3`だったため、最適化
  レベル自体は同一、コンパイラ世代のみが変数)。

全て「フラットにバイナリを置くだけ」から、後にユーザー要望で
`<workload>/src/`(ビルドに使ったソース・Makefile一式)+`<workload>/bin/`
(バイナリ)の構成に再編。canneal/dedupはMakefileがパラメータ化されており
実際のビルドフラグがコマンドライン渡しだったため、`BUILD_COMMAND.txt`に
実行したmakeコマンドを記録。GUPSのMakefileは元が動的リンク版だったため、
静的リンク版に書き換えて格納。

## 4. 重大発見: Sniper同梱Pin 3.11がclone3非対応

GCC15静的ビルド版をコンテナ内のPin(`.bind/pin_kit`、**Pin 3.11**)で
マルチスレッド実行したところ、**スレッドを2つ以上生成した瞬間に必ず
SIGSEGV**することが判明(GUPS/LUで確認)。

原因調査:
- `strace`でネイティブ実行をトレースしたところ、GCC15/glibc 2.41の
  `pthread_create`は新しい`clone3`システムコール(syscall番号435)を使用
  していることを確認(古いstraceは`syscall_435`としか表示できず、rseq
  (syscall 334)も同様に認識できなかった)。
- `GLIBC_TUNABLES=glibc.pthread.rseq=0`でrseq発行自体は止められることを
  確認したが、それでもクラッシュは解消せず → rseqではなく**clone3自体への
  非対応**が真因と特定。
- `.outside_programs/pin-4.2`(2026年版)で同じバイナリを試したところ問題
  なく動作 → Pinのバージョンが原因と確定。

## 5. Sniper本来の推奨Pinバージョンを確認

`.bind/Makefile`(189-227行目)を精査した結果、Sniperのビルドシステムは
実は3パターンのPin取得経路を持つことが判明:
1. `USE_PIN=1` → **Pin 3.22**(2022年、snipersim.org公式配布。ダウンロード
   可能なことを確認済み)
2. `USE_PINPLAY=1` → Pin 3.11ベースのPinPlay(決定論的リプレイ機能用)
3. デフォルト(指定なし) → SDE(Intel Software Development Emulator、
   Pinとは別物)

つまり現行`.bind/pin_kit`のPin 3.11は、`.autodownloaded`マーカーも残って
おらず経緯不明の非標準構成であり、少なくとも「Sniper公式がPin 3.11を推奨
している」わけではないと判明。

**検証**: Pin 3.22(公式ダウンロード)およびPin 3.31(`.outside_programs`
既存)の両方で、GUPS/LU/canneal/dedupの4種(マルチスレッド実行を伴うもの)
全てが素のPin実行下でEXIT:0・正常完了することを確認(LUはVerification
Successfulまで到達)。

## 6. Pin 3.22統合、実施・検証完了

上記5の「未検証の残課題」に着手。SniperのPintool(`sift/recorder`)を
Pin 3.22のヘッダ/ABIで実際にビルドし、`run-sniper`本番パイプラインで
動作確認した。詳しい経緯・遭遇した問題(PinPlayシンボル不足、zlibの
GNU_HASH非互換、静的libzのglibc ABI不整合など)と対処は
`Documents/SniperBugFix.md`の「Pin 3.22統合(2026-07-13)」節を参照。

**結果**: GCC15版バイナリ(`is.S.x`、clone3使用)が`detloc-firsttouch-
v15-pin322`イメージで完走成功(`Verification SUCCESSFUL`)。これにより
「GCC15版バイナリへの移行」が技術的に実現可能と実証できた。BFSは想定
通りPin 3.22でも直らず(別バグと再確認)。本番反映は未実施、要判断。

`binary/Rodinia`をユーザーが誤って削除してしまう事故があったが、git管理
下だったため`git restore`で全1352ファイル(削除のみ、他の変更差分なし)を
安全に復元。復元したlavaMD静的バイナリ(Packed/2TH、`-boxes1d 4`)で
`run-sniper`本番パイプライン経由のPin 3.22テストを実施。

**lavaMDも完走成功**。`Configuration used: cores = 2, boxes1d = 4`・
`[SNIPER] End`まで到達(elapsed 506.93秒)。旧Pin 3.11では`thread-1 to
core-1`(2本目スレッド生成)の瞬間に確実にSIGSEGVしていたポイントを問題
なく突破し、最後まで完走した。これでIS.S(GCC15/clone3)・LU(旧GCC7.3.1)・
lavaMD(旧SIGSEGVパターン)の3種でPin 3.22の健全性を確認できたことになる。

## 7. 教授への説明整理

- **ワークロードのコンパイラ世代差**(gem5=GCC14/15 vs Sniper=GCC4.8/7.3、
  かつNPBバージョンも3.3→3.4で1世代差)は指摘の通り実在した問題であり、
  今回GCC15+NPB3.4への統一で対応。
- **Sniper本体(C++シミュレータ)のビルドコンパイラ**(devtoolset-7/GCC7.3.1)
  は、シミュレート対象コード(ワークロードバイナリ)の最適化度合いとは無関係
  であり、この点は指摘として的外れと説明してよいと整理。
- **未検証の残課題**: 今回動作確認したのは素のPin(pinbin単体)での実行
  のみ。Sniper自前のPintool(`.bind/sift/recorder`、instruction_modeling.cc等、
  Pin 3.11のヘッダ/ABIに対してビルド済み)を実際にPin 3.22/3.31へ差し替えて
  `run-sniper`本番パイプラインで動くかは未検証。ここが解決するまでは
  「対応済み」と言い切るのは早い、という結論で本日はここまで。
