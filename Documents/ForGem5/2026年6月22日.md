## 1. SPO戦略バグ修正

### 問題
`orchestrator.py` でSPOを実行すると `gem5 失敗: Command returned non-zero exit status 1`。

### 原因
`gem5_sim.py` の `compute_active_indices()` に `"SPO"` のケースが存在せず、`ValueError` が発生していた。

### 修正
`gem5_sim.py` に SPO ブランチを追加（暫定実装：Scatter と同じ配置）。

```python
elif strategy == "SPO":
    # スケジューリング優先度協調制御（暫定: Scatter と同じ配置）
    indices = []
    for i in range(num_threads // 2):
        indices.append(i)
        indices.append(i + CORES_PER_NODE)
    return indices[:num_threads]
```

## 8. 先行研究（先輩修士論文）の読み込み

**Jin, Yifan (2022). "Memory-aware Task Mapping for Heterogeneous Multi-Core Systems."
東北大学 大学院情報科学研究科 修士論文.**

### ターゲットシステム
論文のシミュレーション環境は本研究の gem5 構成と**完全一致**：
- 2 NUMA ノード × (Big×4 + Small×4) コア
- Node0: CPU0-3 (Big/2.4GHz) + CPU4-7 (Small/1.2GHz)
- Node1: CPU8-11 (Big) + CPU12-15 (Small)
- シミュレータ：Sniper（本研究では gem5 SE mode に置き換え）

### 2つの優先オプション（MPO / HPO）

| | MPO（メモリ優先） | HPO（異種コア優先） |
|---|---|---|
| 第1ステップ | 通信量でタスクをNUMAノードに集約 | 負荷量でタスクをBig/Smallコアに分類 |
| 第2ステップ | ノード内でBig/Small割り当て | コアタイプ内で通信局所性を最適化 |
| 向き | メモリ強度高・負荷差小 | 負荷差大・メモリ強度低 |

### POSM（切り替え判定式）

```
M = C_mem × (Throughput_comm + Throughput_mem)   # メモリ性 PB
H = C_hetero × CV_load                           # 異種コア性 PB
                                                  # CV_load = σ(負荷) / μ(負荷)
M > H → MPO    M < H → HPO
```

### ベンチマーク別の最適戦略（論文 Fig 4.13/4.15 より）

| ベンチマーク | 負荷差 | メモリ強度 | 最適 |
|---|---|---|---|
| PARSEC-fluidanimate | High | Low | HPO |
| Rodinia-lavaMD | High | Low | HPO |
| NPB-BT | High | High | HPO |
| **NPB-MG** | **Low** | **High** | **MPO** |
| splash2-fft | Low | Low | MPO |

### 本研究との位置づけ

| 項目 | 先輩研究（Jin 2022） | 本研究の拡張方針 |
|---|---|---|
| シミュレータ | Sniper | gem5 SE mode |
| 選択ロジック | 数式ベース POSM（M/H 比較） | AI（階層モデル + コスト感度学習 + Bayesian最適化） |
| ベンチマーク | PARSEC/Rodinia/NPB-BT,MG/splash2 | NPB-CG, NPB-MG（今後拡張） |
| 評価指標 | 実行時間・IPC・エネルギー効率 | 同上 + LOSO（スイート間汎化） |

**NPB-CG は本論文未収録**。スパース行列×ベクトル積でランダムメモリアクセスパターン → メモリ強度高・負荷差小 → **MPO 候補**（要検証）。

## 11. Node0 メモリアクセス偏りの理解

Scatter 戦略でスレッドを左右対称に配置しても Node0 のRead数がわずかに多くなる理由。

### 原因① マスタースレッドのオーバーヘッド

OpenMP はスレッド0がマスター。並列領域の外の逐次処理（初期化・リダクション集計・バリア管理）はすべてスレッド0が担当する。Scatter 配置でも T0 は C0（Node0）であるため、その分だけ Node0 の読み書きが余分に増える。

### 原因② ファーストタッチ

Linux のメモリ割り当ては「最初にそのページを触った NUMA ノードに物理配置」される。OMP スレッド起動前にメインスレッド（T0）が配列を初期化するため、データが Node0 に集中して配置される。並列実行中もそのデータへのアクセスは Node0 のメモリに向かう。

### 研究上の意味

「完全 Scatter でも Node0 偏りが出る」という事実は、後で MPO/HPO の効果を評価するときのベースラインとして重要。スレッド配置だけでなくプログラムの構造的非対称性（マスタースレッドが常に Node0 にいる）がアクセス比率に影響することを示している。
