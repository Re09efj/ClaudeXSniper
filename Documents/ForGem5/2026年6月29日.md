## 1. ランダムフォレストによる NUMA 戦略選択モデルの実装

### 背景

gem5 実験（Class S、4戦略 × 13ワークロード × 4スレッド数 = 208 実行）が完了したため、
「ワークロード特徴量から最適 NUMA 配置戦略を自動選択する」AI モデルの実装に着手した。

### ファイル構成

```
ClaudeX/MachineLearning/
├── RandomForest/random_forest.py   # 学習・評価・可視化スクリプト（全機能）
├── SVM/svm_model.py                # SVM モデル
├── XGBoost/xgboost_model.py        # XGBoost モデル
├── Regression/regression_model.py  # RF 回帰ベースモデル
├── Hierarchical/hierarchical_model.py  # 階層型モデル（RF/SVM/RF+SVM）
├── POSM/posm_baseline.py           # ルールベースベースライン
└── logs/                           # 実行ログ
```

### 設計方針

| 項目 | 選択 | 理由 |
|------|------|------|
| プロファイリング基準実行 | Packed | 全スレッドを Node0 に集約する最保守的な実行。特徴量の baseline として安定 |
| ラベル | `argmin(simSeconds)` または `argmin(energy_J)` | 性能とエネルギー効率の両面を評価 |
| 評価手法 | Leave-One-Out CV | サンプル数が 13（スレッド別）〜52（統合）と少ないため |
| モデル | RandomForestClassifier | n_estimators=200, max_features="sqrt", class_weight="balanced" |

### 特徴量一覧（Packed ランの `stats.txt` から抽出）

| 特徴量 | 内容 |
|--------|------|
| `avg_ipc` | アクティブコアの平均 IPC |
| `ipc_cv` | IPC の変動係数（コア間負荷不均一性） |
| `avg_busy` | 平均 CPU 稼働率（`(numCycles - idleCycles) / numCycles`） |
| `l1d_miss` | L1 dcache ミス率（コア平均） |
| `l2_miss` | L2 ミス率（コア平均） |
| `l3_miss` | L3 ミス率（2ノード加重平均） |
| `l3_intensity` | 命令あたり L3 アクセス数 |
| `mem_intensity` | 命令あたり DRAM アクセス数 |
| `pe_ratio` | P-core / E-core IPC 比（8TH 以上のみ意味あり。2/4TH は 1.0 固定） |
| `num_threads` | スレッド数（ALLTH 統合モデルのみ追加） |

### CPU トポロジと pe_ratio の扱い

```
Node 0: P-core = cpus0-3, E-core = cpus4-7
Node 1: P-core = cpus8-11, E-core = cpus12-15
```

- 2TH / 4TH Packed では cpus0-1 / cpus0-3 のみ使用 → E-core が active にならないため `pe_ratio = 1.0`（均一として扱う）
- 8TH / 16TH では P/E 両方が active → 正常に計算

## 2. スレッド数別モデル評価

### LOO-CV 結果（sim_seconds 基準）

| スレッド | 精度 | 多数決BL | Packed | Scatter | HPO | MPO |
|---------|------|---------|--------|---------|-----|-----|
| 2TH | 23.1% | 69.2% | 9 | 1 | 3 | 0 |
| 4TH | 38.5% | 61.5% | 5 | 0 | 8 | 0 |
| 8TH | 61.5% | 53.8% | 7 | 0 | 5 | 1 |
| 16TH | 92.3% | 92.3% | 12 | 1 | 0 | 0 |

### LOO-CV 結果（energy_j 基準）

| スレッド | 精度 | Packed | Scatter | HPO | MPO |
|---------|------|--------|---------|-----|-----|
| 2TH | 61.5% | 3 | 0 | 2 | 8 |
| 4TH | 38.5% | 2 | 0 | 9 | 2 |
| 8TH | 69.2% | 10 | 0 | 3 | 0 |
| 16TH | 100.0% | 13 | 0 | 0 | 0 |

### 読み取り

- **2TH・4TH**：多数決ベースラインを下回る。クラス不均衡（一方の戦略が圧倒的）の中で `class_weight="balanced"` が過補正する問題
- **8TH**：クラス分布が Packed:7 / HPO:5 / MPO:1 と比較的均衡しており、RF が最も機能する
- **16TH**：12/13 が Packed → 多数決と同率。スレッドが増えると「NUMA 分散の旨みがなくなる」領域に入る
- **スレッド数と最良戦略の傾向**
  - 少ない（2TH）：energy 基準で MPO が優勢（少数スレッドはメモリ帯域分散が効く）
  - 中間（4-8TH）：性能基準で HPO が優勢（P-core 優先が計算効率を上げる）
  - 多い（16TH）：Packed 一択（全コア使い切ると NUMA 分散の効果が消える）

## 3. 全スレッド統合モデル（ALLTH）

### 背景と動機

スレッド別モデルはサンプル数 13 件と少なく LOO-CV が不安定。`num_threads` を特徴量に加えて全スレッド数を統合することで 52 件に増やし、モデルの安定性を改善した。

### LOO-CV 結果

| ラベル | 精度 | サンプル数 | 多数決BL |
|--------|------|-----------|---------|
| sim_seconds | **65.4%** (34/52) | 52 | 63.5% |
| energy_j | **67.3%** (35/52) | 52 | 53.8% |

### 特徴量重要度（LOO-CV 平均）

**sim_seconds 基準:**

| 順位 | 特徴量 | 重要度 |
|------|--------|--------|
| 1 | `avg_busy` | 0.192 |
| 2 | `pe_ratio` | 0.172 |
| 3 | `l3_miss` | 0.114 |
| 4 | `mem_intensity` | 0.110 |
| 5 | `l3_intensity` | 0.100 |
| 10 | `num_threads` | 0.035 |

**energy_j 基準:**

| 順位 | 特徴量 | 重要度 |
|------|--------|--------|
| 1 | `l3_miss` | 0.192 |
| 2 | `num_threads` | 0.151 |
| 3 | `ipc_cv` | 0.119 |
| 4 | `avg_ipc` | 0.102 |
| 10 | `pe_ratio` | 0.038 |

### 解釈

- **性能（simSeconds）** の最適戦略は `avg_busy`（CPU 稼働率）と `pe_ratio`（異種コア効率差）に依存する。スレッド数よりも「どれだけコアが働いているか・P/E コアの効率差がどれほどか」が重要
- **エネルギー（energy_j）** の最適戦略は `l3_miss` と `num_threads` に依存する。L3 を突き抜けて DRAM アクセスが発生する度合いとスレッド数がエネルギー消費を左右する

- GAPBS: S→W で 2倍、S→A で 4倍

GAPBS の W・A スケールは NPB に比べて保守的。A・W の実行時間が出た段階で再評価する。

### 修正したバグ

**XGBoost — `clf_full.fit()` でラベル非連続エラー**

- 原因：特定スレッド数のデータに一部戦略しか存在しない場合、`y_enc` が `[0, 2]` と非連続になる。LOO フォールドはリマップしていたが、フルデータ学習時のリマップが漏れていた。
- 修正：`clf_full.fit()` 前にも `full_unique` で 0-based リマップを適用。

**SVM — `pipe_full.fit()` で 1クラスエラー**

- 原因：データ移動後のパス更新漏れにより 16TH energy_j のデータ読み込みが失敗し、有効サンプルが全て Packed の 1クラスになった。
- 修正：`y.nunique() < 2` の場合は `fit()` をスキップし、重要度をゼロで返す。

**OUTPUTS_DIR パス更新漏れ**

- `Outputs/` → `Outputs/sizeS/` への移動後、XGBoost・SVM の `_rf.OUTPUTS_DIR` 上書きが反映されていなかった。
- 修正：両スクリプトの上書き箇所を `Outputs/sizeS` に修正。

### 特徴量重要度のモデル間差異について

RF で低重要度だった特徴量が SVM/XGBoost でも低重要度になるとは限らない。

- **一致するケース**：特徴量がデータ上で全戦略にわたりほぼ同値 → どのモデルでも判別不能
- **差が出るケース**：SVM（RBF カーネル）は非線形境界を滑らかに捉えるため、木系が見落とした非線形パターンを拾うことがある。また、相関特徴量の扱いがモデルによって異なる（木系は片方を選択、SVM は両方使って距離計算）

3モデルの結果が揃ったら「3モデル全てで低重要度 → 削除候補」「モデル依存で変動大 → 解釈注意」の判断を行う。

## 7. POSM ルールベースベースラインの実装と多手法比較

### POSM (Priority Option Switching Mechanism) とは

Jin (2023) 修士論文で提案されたルールベース戦略選択器。

```
M = mem_intensity  （メモリ帯域要求の代理変数）
H = ipc_cv         （スレッド間負荷不均一性の代理変数）
LOO fold 内で MinMax 正規化し比較:
  M_norm > H_norm → Scatter（高帯域分散戦略）
  H_norm >= M_norm → HPO（高性能コア優先戦略）
```

**特徴と限界：**
- Packed を予測できない（二択しかない）
- 正規化を LOO fold 内で行うことでデータリークを防ぐ
- Class S データでは Packed が多数派（33/52）なので構造的に不利

### 性能比（Performance Ratio）という評価指標

精度（accuracy）だけでは不十分なため、**性能比**を追加導入した。

```
性能比 = mean( sim_seconds[予測戦略] / sim_seconds[oracle最適戦略] )
```

- 1.0 = oracle と完全一致（最適戦略を常に選択）
- 1.05 = oracle より平均 5% 遅い選択をしている
- POSM が精度は低くても性能比が競争力を持つことがあり、この指標で初めて見える

### 全手法 ALLTH 比較結果

※ EPO 追加前（旧 4 戦略: Packed/Scatter/HPO/MPO）の結果。EPO 完了後に再評価予定。

| 手法 | 精度 | 性能比 | 備考 |
|---|---|---|---|
| POSM | 32.7% | 1.0366 | ルールベース、Packed 予測不可 |
| RF分類 | 67.3% | 1.0477 | RandomForestClassifier LOO-CV |
| **SVM** | **78.8%** | **1.0357** | **精度・性能比ともに最良** |
| RF回帰 | 42.3% | 1.0473 | 各戦略の sim_seconds を回帰予測し argmin 選択 |
| RF階層 | 73.1% | 1.0438 | L1=RF(Packed vs non), L2=RF(Scatter/HPO/MPO) |
| SVM階層 | 73.1% | 1.0484 | L1=SVM, L2=SVM |
| RF+SVM階層 | 76.9% | 1.0424 | L1=RF(80.8%), L2=SVM(57.9%) |

**n=52 (Packed:33, HPO:16, Scatter:2, MPO:1)、LOO-CV、Class S ALLTH**

### 逆転現象と考察

精度と性能比の逆転（RF が POSM より精度高いが性能比で劣る）が確認された。

- **原因**：RF が外すとき「Packed を HPO と誤認」などの**コスト大きいミス**をする
- **POSM の強み**：Scatter/HPO の誤差は Packed より小さいことが多い（3戦略の実行時間差が僅か）
- **対策**：コスト感度学習（2026-07-06 以降実装予定）

### 階層型モデルの L1/L2 内訳

| | L1 精度 (Packed vs non) | L2 精度 (Scatter/HPO/MPO) |
|---|---|---|
| RF階層 | **80.8%** | 47.4% |
| SVM階層 | 73.1% | 63.2% |
| RF+SVM階層 | **80.8%** | **57.9%** |

- L1 は RF が優位（Packed の境界が木で捉えやすい）
- L2 は SVM が優位（non-Packed 内の微細な分離は RBF カーネルが向く）
- 組み合わせた RF+SVM階層が最善だが、フラット SVM には届かない（n=19 の L2 データが少なすぎる）

### スレッド数別 性能比サマリー（RF vs SVM）

| スレッド | RF 精度 | SVM 精度 | RF 性能比 | SVM 性能比 |
|---|---|---|---|---|
| 2TH | 53.8% | 61.5% | 1.020 | 1.010 |
| 4TH | 61.5% | 76.9% | 1.043 | 1.024 |
| 8TH | 69.2% | 69.2% | 1.027 | 1.028 |
| 16TH | 92.3% | 92.3% | 1.003 | 1.003 |
| ALLTH | 67.3% | 78.8% | 1.048 | 1.036 |

### 回帰ベースモデルの特徴と失敗ケース

```python
# 各戦略の sim_seconds を RandomForestRegressor で個別予測 → argmin を選択
for s in avail_strategies:
    reg = RandomForestRegressor(n_estimators=200)
    reg.fit(X_train, perf_train[s])
    preds[s] = reg.predict(X_test)[0]
predicted_strategy = min(preds, key=preds.get)
```

- **設計上の優位性**：学習目標が「性能比最小化」と直接一致する
- **2TH での失敗**：n=12 で 4 本の回帰器を学習 → 過学習・汎化不足で性能比 1.047 に悪化
- **ALLTH では改善**（1.047）するが SVM には届かない。データ増加で改善余地あり

## 8. 戦略体系の再設計と EPO 実装

### 旧戦略と廃止理由

| 戦略 | 廃止理由 |
|---|---|
| MPO | Scatter/HPO のエイリアス（ワークロード依存で冗長）。学習データから除外 |
| SPO | Scatter のコピーに過ぎず、SE モードでは実装不能。廃止 |

### 新戦略体系（4戦略）

| 戦略 | CPU マッピング | 概念 |
|---|---|---|
| Packed | Node0 P→E 順に集約 | NUMA 集中、キャッシュ局所性最大化 |
| Scatter | P/E コアをノード間インターリーブ | 帯域分散 |
| HPO | Node0 P → Node1 P → E-core | 高性能コア優先 |
| **EPO** | Node0 E → Node1 E → P-core | **省電力コア優先（HPO の逆）** |

```
NUMA トポロジ:
  Node 0: CPU  0- 3 (P-core),  4- 7 (E-core)
  Node 1: CPU  8-11 (P-core), 12-15 (E-core)

HPO cpu_map: [0,1,2,3, 8,9,10,11, 4,5,6,7, 12,13,14,15]
EPO cpu_map: [4,5,6,7, 12,13,14,15, 0,1,2,3, 8,9,10,11]
```

**設計の対称性：**
- 配置軸：Packed (集中) vs Scatter (分散)
- コア軸：HPO (性能優先) vs EPO (電力優先)

### 実装した変更箇所

| ファイル | 変更内容 |
|---|---|
| `utility/cpu_affinity.py` | EPO の cpu_map・STRATEGY_DESC を追加 |
| `ClaudeX/gem5_sim.py` | `compute_active_indices()` に EPO ブランチを追加 |
| `orchestrator.py` | `--strategies` / `--workers` CLI 引数を追加 |
| `MachineLearning/*/` | STRATEGIES を `["Packed","Scatter","HPO","MPO"]` → `["Packed","Scatter","HPO","EPO"]` に変更（全5モデル） |
| `XGBoost/xgboost_model.py` | `n_jobs=6` を全3箇所の XGBClassifier に追加（リソース消費の抑制） |

## 9. ML 自動学習キューの構築・起動

### 概要

EPO シミュレーション完了を自動検知し、旧 ML 出力を破棄した上で全モデルを順次再学習するキュースクリプトを作成・起動した。

```bash
nohup bash ClaudeX/run_ml_after_epo.sh > ClaudeX/MachineLearning/logs/ml_queue_run.txt 2>&1 &
# PID=1293888
```

### 実行順序

```
EPO ログ監視（60秒ポーリング）
  ↓ 「EPO Class S 全スレッド数完了」検知
POSM → RF → Regression → SVM → Hierarchical → XGBoost
  ↓ 各ステップ失敗時は即停止
```

- 各モデルのログは個別 `.txt` ファイルに保存
- XGBoost は `n_jobs=6` に制限済み（gem5 Class A と共存）
- 旧 ML 出力ファイル（CSV・PNG 約160ファイル）は EPO 追加により別物になるため削除済み
