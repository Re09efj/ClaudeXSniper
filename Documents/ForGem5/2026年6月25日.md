## 1. 解析的電力モデルへの切り替え（`mcpat_bridge.py`）

### 背景

電力データ取得のため McPAT（v1.3）をインストールして試みたが、以下の問題で断念した。

| 問題 | 詳細 |
|---|---|
| **実行時間** | Xeon.xml リファレンスで15分超。`homogeneous_cores=0` の16コア構成では16倍起動される |
| **ヘテロ構成の不正確さ** | McPAT の XML では P/E コアを区別できず、全コアに同一 OOO パラメータ（`issue_width=8`, `ROB=192`）が適用される |
| **プロセス設定の矛盾** | 22nm（Haswell世代）+ E-core 1GHz という組み合わせは実在しない |

### 解決方針

McPAT を完全に廃し、**gem5 のネイティブ統計から直接計算する解析的モデル**のみを実装した。

### 解析的電力モデルの実装

#### 電力定数

```python
_P = dict(
    p_core_dynamic_W   = 8.0,    # P-core ピーク動的電力
    p_core_leakage_W   = 0.5,    # P-core リーク電力
    e_core_dynamic_W   = 1.5,    # E-core ピーク動的電力（P の 1/5）
    e_core_leakage_W   = 0.2,    # E-core リーク電力
    l2_access_energy_J = 0.5e-12,
    l3_access_energy_J = 2.0e-12,
)
```

絶対値は仮想システムの設定値。研究目的は戦略間の**相対的なエネルギー効率比較**なので、比率の妥当性（P-core は E-core の5倍電力）が重要。

#### アクティビティ係数

```python
# IPC ベース（旧）: ストール中のコアの電力を無視 → 過小評価
activity = cpu["ipc"] / peak_ipc   # → ~30%

# idleCycles ベース（現）: gem5 が計算済みのアイドルサイクルを利用
busy_cycles = max(num_cycles - idle_cycles, 0)
activity    = busy_cycles / num_cycles   # → ~98%（実態を反映）
```

#### 電力計算式

```
P_core    = Σ (activity × TDP) + Σ leakage
P_cache   = (L2_accesses × 0.5pJ + L3_accesses × 2pJ) / sim_s
P_dram    = Σ dram.rank0.totalEnergy [pJ] × 1e-12 / sim_s  ← gem5 計算値
total_W   = P_core + P_cache + P_dram
energy_J  = total_W × sim_s
```

#### 結果の書き込み先

`stats.txt` / `stats.csv` に直接追記することで既存の解析パイプラインがそのまま電力データを読める。

| キー | 内容 |
|---|---|
| `power.total_W` | 全体電力（W） |
| `power.dynamic_W` | コア＋キャッシュ動的電力（W） |
| `power.leakage_W` | コアリーク電力（W） |
| `power.dram_W` | DRAM電力（gem5 DRAMInterface 計算値）（W） |
| `power.energy_J` | 消費エネルギー（J） |

重複書き込み防止のため `power.total_W` が既存の場合はスキップ。

## 2. 戦略別バイナリ廃止・環境変数方式への移行

### 実行チェーン（移行後）

```
orchestrator → get_cpu_map() → "0,1,2,3,..."
             → gem5_sim.py --cpu-map=0,1,2,3,...
             → プロセス環境変数 NUMA_CPU_MAP=0,1,2,3,...
             → numa_affinity.c が getenv() で読み取る
```

### `numa_affinity.c` の変更（NPB 全7本）

```c
static void load_cpu_map(void) {
    const char *env = getenv("NUMA_CPU_MAP");
    if (!env) return;
    // カンマ区切りで16個の整数をパース
}
void set_affinity_(int *thread_id_ptr) {
    load_cpu_map();
    CPU_SET(cpu_map[*thread_id_ptr], &cpuset);
    sched_setaffinity(0, ...);
}
```

- Fortran 呼び出し版（BT/CG/MG/EP/FT/SP）: `set_affinity_(int*)`
- C 呼び出し版（IS）: `set_affinity(int)`
- lavaMD の `numa_affinity_c.c` はすでに env var 対応済みのため変更不要

### `cpu_affinity.py` のクリーンアップ

- `patch_cpu_map()` / `compile_strategy()` / `_affinity_c_path()` を削除
- 戦略別バイナリ（`bt.S.Packed.x` 等）を全削除、クラス別ベースバイナリのみ残存

## 3. GAPBS 追加対応

### GAPBS（GAP Benchmark Suite）とは

UC Berkeley の Scott Beamer らが 2015年に発表したグラフ処理ベンチマーク集（OpenMP C++）。不規則メモリアクセスが特徴で NUMA 効果を強く引き出す。

| 略称 | アルゴリズム |
|---|---|
| BFS | Breadth-First Search |
| PR | PageRank |
| BC | Betweenness Centrality |
| CC | Connected Components |
| SSSP | Single-Source Shortest Paths |
| TC | Triangle Counting |

### NUMA affinity の組み込み（GAPBS）

`src/numa_affinity.h` を新規作成し、`src/benchmark.h` からインクルード。`BenchmarkKernel()` 冒頭で全スレッドの affinity を設定。`benchmark.h` は全6カーネルから共通インクルードされるため1ファイルの変更で全アルゴリズムに反映される。

### ベンチマーククラスの統一定義

NPB の `CLASS=S/W/A/B/C/D` を共通キーとして各スイートのサイズに変換する。

| CLASS | NPB | GAPBS (`-g` 値・頂点数) |
|---|---|---|
| S | Class S | `-g 10`（1,024頂点） |
| W | Class W | `-g 11`（2,048頂点） |
| A | Class A | `-g 12`（4,096頂点） |
| B | Class B | `-g 13`（8,192頂点） |
| C | Class C | `-g 14`（16,384頂点） |
| D | Class D | `-g 16`（65,536頂点） |

GAPBS のスケール値は gem5 実行時間を考慮した値（実ハードウェアの標準値 `-g 20`〜`-g 27` より大幅に小さい）。初回実験後に調整予定。

### MPO マップの追加

GAPBS は不規則アクセスで両ノードの帯域が必要なため、全アルゴリズムの MPO マップは Scatter（インターリーブ）と同一。

```python
"BFS": [0, 8, 1, 9, 2, 10, 3, 11, 4, 12, 5, 13, 6, 14, 7, 15],
# PR, BC, CC, SSSP, TC も同様
```

### PARSEC の廃止判断

PARSEC 3.0（2011年）は設計が古く NUMA・異種コアを想定していない。入力ファイルも別途取得が必要で手間がかかる。GAPBS と NPB に集中する方針とし、実験対象から除外した（コードは残存）。

## 4. Jin論文の Homo/Hetero 定義の再確認・設計方針修正

### Jin論文の定義

TABLE 4.1/4.2 を精読した結果、Homo/Hetero の違いは「配置戦略の違い」ではなく「**CPU の動作周波数が均一か混在かの違い**」であった。

| | Homogeneous | Heterogeneous |
|---|---|---|
| NUMAノード数 | 1 | 1 |
| コア構成 | 16× 2.4GHz（全コア同速） | 8× 2.4GHz + 8× 1.2GHz（混在） |

### キャッシュ構成比較

| パラメータ | Jin論文 | 私たちの設定 |
|---|---|---|
| NUMAノード数 | 1 | 2（研究目的で意図的に異なる） |
| L1 I/D | 256KB / 8-way | 256KiB / 8-way ✅ |
| L2 | 2MB / 8-way | 2MiB / 8-way ✅ |
| L3 | 8MB / 16-way | **512KiB** / 16-way ❌（意図的縮小） |

L3 を 512KiB に縮小しているのは、ワーキングセットが L3 に収まらないようにして DRAM アクセスを頻発させ、NUMA ペナルティを顕在化させるため（L3 ミス率 = 100%）。

### ホモジニアスモード不要の判断

研究テーマは「異種 NUMA システム上での負荷分散戦略の自動設計」であり、「異種 vs 同種どちらが速いか」は問いではない。同種比較は Jin 論文を引用すれば済む。

→ `HOMO_STRATEGY` / `HETERO_STRATEGY` の概念を廃止。比較は常に全戦略（Packed/Scatter/HPO/MPO）を対象とする。

### Jin論文の EnergyEfficiency 定義

```
EnergyEfficiency = IPC / EnergyConsumption
```

これにより電力推定（Section 1 の解析的モデル）が研究に必要であることを確認。

## 5. latency_breakdown バグ修正・NUMA レイテンシ設定の確認

### バグ：L3 Stats not available

`generate_latency_graph()` が `system.l3cache{n}.overallHits::total` を探していたが、**gem5 はこの stat 名を出力しない**。

```python
# 修正前（stat が存在しない）
hits   = get(f"system.l3cache{n}.overallHits::total")
misses = get(f"system.l3cache{n}.overallMisses::total")
total  = hits + misses

# 修正後
accesses = get(f"system.l3cache{n}.overallAccesses::total")
misses   = get(f"system.l3cache{n}.overallMisses::total")
hits     = accesses - misses
total    = accesses
```

### L3 ミス率 100% の意味

BT_S_Scatter の stats：`overallMissRate::total = 1`（100%）。512KiB L3 にワーキングセットが収まらず、L3 が素通りになっている。意図的な設計（`gem5_sim.py` の `L3_SIZE = "512KiB"` コメント通り）。

### NUMA アクセスペナルティの設定値

```python
# gem5_sim.py
NUMA_INTERCONNECT_LATENCY = 50  # cycles @ 1GHz = 50ns
system.membus = SystemXBar(frontend_latency=50, forward_latency=50, response_latency=50)
```

実際のアクセスレイテンシ内訳：

| 経路 | 合計 |
|---|---|
| L1 ヒット | ~2ns |
| L2 ヒット | ~12ns |
| L3 ヒット | ~32ns |
| DRAM（L3 ミス） | ~150ns 以上 |

実測値（BT Scatter）：`l3cache0.overallAvgMissLatency = 107,840 ticks ≈ 108ns`

**注意**：`mem_ctrl0`（Node0）と `mem_ctrl1`（Node1）は同じ `membus` に接続されているため、ローカル/リモート DRAM で明示的なレイテンシ差はない。NUMA ペナルティは first-touch によるデータ配置と L3 キャッシュ局所性によって**暗黙的**に生まれる。
