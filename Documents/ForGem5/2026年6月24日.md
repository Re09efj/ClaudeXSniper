## 2. NPB 追加ベンチマーク対応（EP / IS / FT / SP）

### 背景

既存の NPB OMP ディレクトリには BT・CG・MG しか NUMA affinity 対応バイナリがなかった。残る EP・IS・FT・SP もソースは存在するため、同じ仕組みを整備して全5戦略バイナリを生成した。

### 2-1. NUMA affinity の仕組みおさらい

既存の実装パターン：
1. 各ベンチマークディレクトリに `numa_affinity.c` を配置
2. `static int cpu_map[16]` に戦略ごとの CPU マッピングを記述
3. Fortran から `call set_affinity(myid)` で呼べるよう `set_affinity_()` を実装（gfortran は末尾アンダースコア規約）
4. OMP 並列セクション冒頭で各スレッドが自分の `myid` を渡して `sched_setaffinity` を発行
5. gem5 の `schedSetaffinityFunc` がこれを横取りし `process->currentNumaNode` を更新

IS だけ C ベンチマークのため `set_affinity(int)` で直接呼ぶ（ポインタ渡し不要）。

### 2-2. 各ベンチマークへの対応内容

#### SP（`SP/initialize.f90`）

`initialize.f90` の `!$omp parallel` ブロックに affinity 宣言と呼び出しを追加。

```fortran
!$    integer           omp_get_thread_num
!$    external          omp_get_thread_num
!$    external          set_affinity
      integer           myid

!$omp parallel default(shared)  &
!$omp& private(i,j,k,m,...,myid)
!$    myid = omp_get_thread_num()
!$    call set_affinity(myid)
```

#### EP（`EP/ep.f90`）

主要並列ブロック（`!$omp parallel default(shared) reduction(+:sx,sy)`）の冒頭に追加。`myid` を private リストに加えた。

```fortran
!$omp parallel default(shared) reduction(+:sx,sy)  &
!$omp&  private(k,kk,t1,t2,t3,t4,i,ik,x1,x2,l,myid)
!$    myid = omp_get_thread_num()
!$    call set_affinity(myid)
```

#### FT（`FT/ft.f90`）

FT は `!$omp parallel do` のみで明示的な `myid` 取得箇所がないため、専用の `init_affinity` サブルーチンをファイル末尾に追加し、`timer_start(T_total)` の直後から呼ぶようにした。

```fortran
      call timer_start(T_total)
      call init_affinity()          ! ← 追加

! ft.f90 末尾に追加
      subroutine init_affinity
!$    integer           omp_get_thread_num
!$    external          omp_get_thread_num
!$    external          set_affinity
      integer           myid
!$omp parallel default(shared) private(myid)
!$    myid = omp_get_thread_num()
!$    call set_affinity(myid)
!$omp end parallel
      return
      end
```

#### IS（`IS/is.c`）

C ベンチマークのため Fortran 規約（末尾アンダースコア）不要。  
`is.c` 先頭に前方宣言を追加し、最初の `#pragma omp parallel` ブロック内で呼び出す。

```c
// 先頭に追加
void set_affinity(int thread_id);

// 最初の並列ブロック内
myid = omp_get_thread_num();
num_threads = omp_get_num_threads();
set_affinity(myid);     // ← 追加
```

`IS/numa_affinity.c` は C インターフェース版（`void set_affinity(int thread_id)`）として新規作成。

### 2-3. Makefile 修正（全4ベンチマーク共通）

各 Makefile の `OBJS` に `numa_affinity.o` を追加し、依存ルールを記述。  
Fortran 系は make の組み込み `.c.o` 暗黙ルールで自動コンパイル（`CCOMPILE = gcc -c -O3 -fopenmp -static`）。

```makefile
OBJS = ... \
       numa_affinity.o \
       ...

numa_affinity.o:  numa_affinity.c
```

### 2-4. MPO マップの追加（`cpu_affinity.py`）

`MPO_MAPS` 辞書に EP・IS・FT・SP のワークロード特性に基づいたマッピングを追加。

| ベンチマーク | MPO 配置方針 | 根拠 |
|---|---|---|
| **EP** | HPO と同じ（Node0 P-core 優先） | スレッド間データ共有なし。各スレッドが独立した乱数列を処理するため first-touch で自ノードに確保される |
| **IS** | Node0 先詰め | バケットソートで全スレッドが `bucket_size[]` を共有読み書き。first-touch で Node0 に確保されるため Node0 先詰めで遠隔アクセスを削減 |
| **FT** | Scatter（インターリーブ） | グローバル転置が発生するため両ノードに帯域分散が有利 |
| **SP** | Node0 先詰め（BT と同じ） | 全スレッドが共有配列を高頻度アクセスする三重対角ソルバ |

## 3. ベンチマークスイート拡充（ソース取得）

### 3-1. 方針決定

SPLASH2 は 2000 年代前半のベンチマークで現代アーキテクチャへの適合性が低いため採用見送り。代わりに最新の論文で頻用されている3スイートを選定した。

| スイート | 選定理由 |
|---|---|
| **PARSEC 3.0** | Jin 修士論文でも使用。実用アプリ模倣の pthreads ベンチマーク群。NUMA 研究で最も引用が多い |
| **GAPBS** | グラフ解析（BFS, PageRank, SSSP, BC 等）。不規則メモリアクセスが NUMA 効果を強く引き出す。2020年以降の論文で急増 |
| **Rodinia** | lavaMD を含む HPC 系ベンチマーク群。OpenMP 版を取得。GPU/CPU 両対応で多様性あり |

### 3-2. ダウンロード方法

git HTTPS クローンが環境変数 `SSH_ASKPASS=/usr/bin/ksshaskpass` により認証ダイアログを要求して失敗する問題が発生。  
`GIT_ASKPASS=echo` を付加することで公開リポジトリへの匿名アクセスを強制し解決した。

```bash
GIT_ASKPASS=echo git clone --depth=1 https://github.com/<repo>.git <path>
```

### 3-3. 取得内容

**GAPBS**（`binary/GAPBS/`）

- OpenMP ベース。`make` でそのままビルド可能

- 既存の `binary/lavaMD/` は本スイートの独立コピーであることが判明（将来的に Rodinia 版に統合可能）

**PARSEC 3.0**（`binary/PARSEC/`）
- ソース: `https://github.com/cirosantilli/parsec-benchmark`（Princeton 公式 GitHub ミラー）
- 収録アプリ: blackscholes, bodytrack, facesim, ferret, fluidanimate, freqmine, raytrace, swaptions, vips, x264
- 収録カーネル: canneal, dedup, streamcluster

## 4. スレッド移動（Migration）に関する技術的考察

### 問題提起

「実行中に戦略を変えたい（最初の割り振りだけでなく移動も実装したい）」という要件について、gem5 改造とベンチマーク改造のどちらが適切か検討した。

### 移動の2層構造

| 層 | 内容 | 現在の実装状態 |
|---|---|---|
| **メモリ affinity** | 新規ページをどのノードから確保するか | ✅ 実装済み（`currentNumaNode`） |
| **スレッド配置** | スレッドがどの CPU コアで走るか（→ どの L3・メモリコントローラを使うか） | ❌ 未実装 |

真の移動には両方が必要。現在の `sched_setaffinity` 実装は「メモリ affinity のみ」を変えているため、スレッドが走るコアは変わっていない。

### gem5 改造 vs ベンチマーク改造

**gem5 改造が正解**。理由：

- ベンチマーク改造は全スイート・全ベンチマークへの個別対応が必要（13種 × 3スイート規模）
- gem5 側に tick ベースの移動コールバックを持てばすべてのベンチマークに一律適用できる
- gem5 の `ThreadContext::takeOverFrom()` を利用した CPU 間コンテキスト移動は既に CPU モデル切り替えで実績あり

### 実装の選択肢

| 方法 | 内容 | 難易度 |
|---|---|---|
| **A: メモリ affinity のみ変更** | gem5 の tick コールバックで `currentNumaNode` を更新。既存ページは変わらず | 低 |
| **B: 完全移動（ThreadContext 移動）** | `takeOverFrom()` でスレッドを別 CPU に移す。パイプライン状態の保存・復元が必要 | 高 |
| **C: バリア点での再 affinity** | OpenMP バリア後に `sched_setaffinity` を再呼び出し。gem5 がそれを拾って両層を更新 | 中（両側変更が必要） |

### 推奨方針

まず方法 A（メモリ affinity 変更のみ）で効果を計測してから、必要に応じて方法 B（ThreadContext 移動）に進む段階的アプローチが安全。
