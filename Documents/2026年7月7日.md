## 2. 半教師あり学習(Label Propagation/Spreading)の検証 → 失敗

「特徴量だけ学習し、正解ラベルが必要なのは一部だけでいい」という都合の良い
枠組みとして、Label Propagation/Spreading(グラフ全体にラベルを拡散する
半教師あり学習)を提案・検証した。

既存の二段階k-NN Nested CV(43グループ、NPB系のみ、leave-one-workload-out)
と同じ枠組みに載せ替えて比較:

```
alpha=0.05: 32.6%
alpha=0.5:  37.2%
alpha=0.8:  39.5%
alpha=0.95: 41.9%(頭打ち)
alpha=0.99: 39.5%

(参考) 多数派ベースライン: 46.5%
(参考) 二段階k-NN(NetLSD, 既存最良): 51.2%
```

alphaを上げるほど改善する傾向だったが、41.9%で頭打ちとなり多数派ベースラインにも
届かず。少数派クラス(EPO等)が極端に少ないこの規模のデータでは、グラフ全体を
グローバルに平滑化する手法より、k-NNのようなローカルな判断の方が頑健、という
7/6の教訓が再確認された。

## 3. 入力をmem_access.csvに変える案の検証 → 失敗

「通信行列(comm.csv)ではなくmem_access.csv(スレッドごとのread/write回数・
バイト数)を入力にしてはどうか」という提案を検証した。

mem_access.csvはtid,n_reads,n_writes,sz_reads,sz_writesという**スレッドごとの
1次元ベクトル**であり、comm.csvのようなスレッド×スレッドのグラフではないため、
NetLSD(グラフラプラシアンのスペクトル)はそのままでは適用できない。代わりに
各指標の分布統計量(平均・変動係数・max/mean比・Gini係数)でsize-invariantな
記述子を作り、同じ二段階k-NN Nested CVで検証:

```
mem_access 未正規化(絶対量込み): 34.9%
mem_access 正規化(分布の形のみ): 37.2%

(参考) comm行列NetLSD: 51.2%
(参考) 多数派ベースライン: 46.5%
```

どちらも多数派ベースラインにすら届かず。comm.csv(構造情報: 誰と誰が通信するか)と
mem_access.csv(負荷情報: 各スレッドが独立にどれだけ読み書きしたか)は本質的に
違う情報を持ち、配置戦略(Packed/Scatter等)の巧拙を当てる今回のタスクには
構造情報が必要、という仮説が消去法で裏付けられた。

## 4. NetLSD(構造)+mem_access(負荷)の特徴量連結 → 失敗(5回目の「合体は損」)

念のため両者を連結した特徴量でも検証:

```
NetLSDのみ(構造):        51.2%
mem_accessのみ(負荷):    34.9%
連結(構造+負荷):         34.9%(NetLSD単体より悪化)
```

7/6の数式×NetLSD合体4パターン全滅に続き、5回目の「合体は損」の再現。
低品質な特徴量を混ぜるとユークリッド距離が歪み、良い特徴量単体の近傍選択が
狂うという次元の呪いの典型パターンが、入力の組み合わせを変えても一貫して
観測された。

## 5. 重大なバグ発見: TH=2でNetLSD記述子が全ワークロードで完全に同一値に退化

能動学習ランキングを実装する過程で発覚。ラベル付き43グループのうちTH=2の
グループ(BT/CG/FT/MG/SP等、S/W両クラス)を調べたところ、NetLSD記述子が
**全て`[1,1,1,1,1,...]`という同一値**になっていた。

原因: 一部のTH=2の`comm.csv`は対角成分のみで非対角(スレッド間の実通信)が0。
標準的な組合せラプラシアン`L = D - W`は自己ループ(対角成分)が数式上
完全にキャンセルする性質があるため、非対角が全部0だとラプラシアンがゼロ行列
になり、熱核トレース記述子が常に無情報な`[1,1,...,1]`になる。

これまでのNested CV 51.2%という結果は、実質的にTH=8/12/16のデータが支えており、
TH=2のグループは(全ワークロードで区別がつかないため)k-NNの近傍選択に無意味な
ノイズとして混ざっていた可能性がある。**未修正のまま持ち越し**。優先度は低いと
ユーザー判断(実測待ちの間の作業として着手可能)。
