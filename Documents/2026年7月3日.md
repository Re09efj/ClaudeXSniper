## 1. sizeA タイムアウト機構の原因分析

### 発覚した問題
- `logs/timeoutwl.log` に BT/SP (Class A, 2TH) が2回のリトライとも `elapsed=43201s > timeout=43200s`（12時間）で強制終了されていた
- `orchestrator.py` のタイムアウトは `timeout_sec = expected_sec × 3`
- `expected_sec` は `utility/run_profile.py` の `_MAX_ESTIMATED_SEC = 14400s`（4時間）で頭打ちされる仕様
- BT は `BT_W_2=6175s × 40.0倍`、SP は `SP_W_2=14094s × 180.0倍` という生の推定値（それぞれ68.6時間・約29日相当）が出ていたが、キャップのせいでどちらも一律 `timeout=43200s` に切り詰められていた
- MG（`W×90.0`、実測がたまたま12時間以内に収まっていた）は成功しており、BT/SP だけが「たまたま間に合わなかった」形

### 分かったこと
- タイムアウト機構自体は Pin (`sift_recorder`) の SIGSEGV 対策ではない。それは7/2に `-O3→-O2` 再ビルドで別途解決済み
- 7/2の作業ログに、まさに今回のケース向けに `--no-timeout` フラグと「BT/MG/SP はこれで放置する」という方針が**すでに実装・決定済み**だった
- しかし実際に動いていたジョブは `--no-timeout` を付けずに起動されていたため、想定通りタイムアウトに巻き込まれていたのが直接原因

### 対応
- 死んでいた古い BT/SP フォルダ（`20260702_214146` 分・8個）を削除
- `--threads 2 --bench-class A --workloads BT SP --no-timeout` で再実行 → ユーザーが後で強制停止
- 強制停止に伴い `20260703_172704` 分の中断フォルダも削除
- **確認**: Outputs/sizeA/2TH で完了済みなのは BT・SP 以外の12ワークロード×4戦略＝48実験のみ

## 2. 「軽い組」「重い組」の並列実行に分割

### 問題
- `orchestrator.py` は `--threads` に複数値を渡しても内部で逐次ループするため、BT/SP のような激重ワークロードが1つのスレッド数に混ざっていると、次のスレッド数（6TH等）が丸ごと待たされる

### 対応
- 重い組（BT, MG, SP）と軽い組（残り10ワークロード）を**別プロセス**として並列起動
- Sniper の計測値はシミュレータ内部の仮想時間ベースなので、ホストCPUの競合があっても計測結果自体の正しさには影響しない（完了までの実時間が伸びるだけ）と判断

```bash
# 重い組: BT/MG/SP を全スレッド数で逐次実行（タイムアウトなし）
nohup python3 orchestrator.py --threads 2 6 8 12 --bench-class A \
  --workloads BT MG SP --no-timeout >> logs/heavy.log 2>&1 &

# 軽い組: それ以外のワークロードを全スレッド数で逐次実行（重い組を待たない）
nohup python3 orchestrator.py --threads 2 6 8 12 --bench-class A \
  --workloads CG FT IS lavaMD BFS PR BC CC SSSP TC >> logs/light.log 2>&1 &
```

## 4. TabPFN の精度比較・Oracle比（perf_ratio）指標の追加

### ALLTH_SW（sizeS+W合算, n=104）での比較
| モデル | 精度 |
|---|---|
| RandomForest | 85.6% |
| SVM | 86.5% |
| XGBoost | 89.4% |
| TabPFN | 93.3%（後述の理由で再現せず） |

### スレッド別（sizeS単独, n=13）での比較
| スレッド数 | SVM | XGBoost | TabPFN |
|---|---|---|---|
| 2TH | 69.2% | 69.2% | 61.5%（最下位） |
| 4TH | 69.2% | 69.2% | 69.2% |
| 8TH | 100% | 100% | 100% |
| 16TH | 92.3% | 61.5% | 92.3% |

→ サンプル数が少ない単一スレッドでは TabPFN の優位性はほぼ出ない。ALLTH_SW のようにサンプルを束ねたときに「小サンプルに強い」という TabPFN の本領が発揮される。

### Optuna との関係
- 現状 `Optuna/optuna_hier.py` は RF/SVM/LogReg による階層型分類器の構造探索に使っているのみで、TabPFN には未適用
- TabPFN は事前学習済みTransformerによる in-context learning が前提のため、RF の `n_estimators` のような従来型ハイパーパラメータをそもそも持たず、Optuna 探索にはあまり向かない

### Oracle比（perf_ratio）の実装
- `Regression/comparison.csv` 等にあった「Oracle比 = 予測戦略のsim_seconds ÷ 最速戦略のsim_seconds（1.0=完全一致）」を TabPFN にも追加
- `MachineLearning/TabPFN/tabpfn_model.py` に `perf_ratio()` を追加し、`run()` で計算・`performance_{label}.csv` に保存
- `MachineLearning/train_combined.py` に `build_allth_perf()` / `perf_ratio()` を追加し、`run_tabpfn()` で ALLTH_SW 版も計算するよう変更

#### 結果（2/6/8/12TH, sizeS単独 n=13）
| スレッド数 | 精度 | Oracle比 |
|---|---|---|
| 2TH | 61.5% | 1.0184 |
| 6TH | 100% | 1.0013 |
| 8TH | 100% | 1.0005 |
| 12TH | 61.5% | 1.0135 |

精度が61.5%まで落ちるスレッド数でも、Oracle比は1.4〜1.8%悪化に留まっており、「ラベルを外しても実行時間的にはそこまで悪化しない」傾向はPOSMだけでなくTabPFNでも見られる。

#### ALLTH_SW（sizeS+W合算, n=104）
| モデル | 精度 | Oracle比 |
|---|---|---|
| POSM | 5.8%（n=52） | 1.0075 |
| RF分類 | 69.2%（n=52） | 1.1328 |
| SVM | 75.0%（n=52） | 1.0245 |
| RF回帰 | 36.5%（n=52） | 1.0536 |
| **TabPFN** | 86.5%（n=104） | **1.0029（最良）** |

### 重要な発見：TabPFN の非決定性
- ALLTH_SW を2回実行したところ、精度が **93.3% → 86.5%** に変化した（Oracle比は1.0029で安定）
- `TabPFNClassifier()` に `random_state` を指定していないため、クラウドAPI側の推論に非決定性がある可能性
- RF/SVM/XGBoost は `random_state=42` 固定で毎回同一結果になるが、TabPFN のみこの再現性の問題がある
- 論文等に書く場合は複数回実行して平均・分散を示すなどの対策が必要
