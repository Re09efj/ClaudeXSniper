## 1. ML 特徴量の整理（28次元への統一）

### 発覚した問題
- `random_forest.py` が独自の `parse_stats()` と `FEATURE_COLS`（9特徴量）を内部に持っており、
  SVM・XGBoost も `random_forest` から import していたため、全モデルが 9 特徴量で動いていた
- `ml_utils.py` に追加した 19 特徴量が実際には使われていなかった

### 対処
- `random_forest.py` の独自 `parse_stats()` / `FEATURE_COLS` を削除
- `ml_utils.py` の `parse_stats()` / `FEATURE_COLS`（28特徴量）を一元的に使用するよう統一

### 現在の特徴量（28本）
| カテゴリ | 特徴量 |
|---|---|
| IPC・コア | avg_ipc, ipc_cv, pe_ratio |
| NUMA | local_ratio, remote_ratio |
| メモリ階層 | l2_intensity, l3_intensity, mem_intensity, read_ratio |
| CPI内訳 | cpi_l3_ratio, cpi_dram_local_ratio, cpi_dram_remote_ratio, cpi_branch_ratio, cpi_sync_ratio |
| μop | uop_load_ratio, uop_store_ratio, uop_fp_ratio |
| TLB・分岐 | dtlb_miss_rate, branch_mispredict_rate, itlb_miss_rate, stlb_miss_rate |
| DRAM | avg_dram_latency, dram_queue_ratio |
| L1D | l1d_hit_ratio, load_l1_ratio |
| 同期 | futex_wake_per_minst |
| 帯域・Roofline | mem_bw_mbps, operational_intensity |

### 削除した指標（全スレッドで重要度 0.000）
- `mutex_lock_per_minst` / `barrier_wait_per_minst` / `mutex_contention_ratio`
- 理由：OpenMP ワークロードは pthread_mutex・barrier を直接使わず全ゼロ

## 4. 特徴量増加で精度が下がった件の考察

### 9特徴量 → 28特徴量で RF ALLTH が 82.7% → 69.2% に低下

**個別 TH（2TH 等）での次元過多**
- 12 サンプル / 28 特徴量 = 比率 0.43（特徴量の方が多い）
- RF は高次元小サンプルで過学習しやすい

**ALLTH（52サンプル）での低下の主因**
1. 元の 9 特徴量は NUMA 配置に直接効く指標を意図的に厳選していた可能性
2. 相関する特徴量（`mem_bw_mbps` と `mem_intensity`、`itlb_miss_rate` と `dtlb_miss_rate` 等）が増えると RF の分岐が分散し重要な特徴量が使われにくくなる
3. SVM はカーネル空間で距離を測るため高次元への耐性が高く ALLTH 84.6% を維持

**SVM は 28 特徴量でも安定 → ALLTH 推薦モデルは SVM**

### 今後の方針候補

- EDP（E×T）ラベルで省電力最適化モデルを構築
