## 4. HPO 惨敗の考察とスレッド構成の見直し

### HPO がEPOに負ける理由
- HPO = P コア優先配置 → P コアに負荷が偏り、E コアが遊ぶ
- Sniper のエネルギーモデルは動作周波数ベースのため、実行時間が短くても消費エネルギーが高くなりやすい
- ワークロードの実装が OpenMP スレッドを均一に使うため、「P コア優先」の意味が薄れる場面がある
- **HPO が劣っているのではなく実装依存**

### スレッド数の変更
- 16TH は戦略差が出にくい（ほぼ全コア使用で Packed≒Scatter≒HPO≒EPO）→ **廃止**
- **12TH を採用**（P×4 + E×8 など戦略差が出やすい構成）
- 最終的なスレッドラインナップ：**[2, 6, 8, 12] TH**

## 7. 12TH BFS/HPO SIGSEGV → sift_recorder を -O2 で再ビルド

### 原因
- sift_recorder（Pin ベースの SIFT レコーダ）が `-O3` でビルドされており、12TH BFS の HPO 戦略で SIGSEGV が発生
- 過去（sizeS）でも同様の問題が発生し `-O2` で解決していたが、Podman イメージが更新されていなかった

### 対処：`rebuild_sift_o2.sh` の作成と実行

```
ClaudeXSniper/rebuild_sift_o2.sh
```

- Podman イメージ `snipersim/snipersim:latest` から一時コンテナを起動
- `pin_kit/source/tools/Config/makefile.unix.config` の `-O3` を `-O2` に置換
- sift_recorder をクリーンビルド
- `podman commit` でイメージを更新
- `trap cleanup EXIT` で終了時に一時コンテナを自動削除

### 落とし穴
- `podman commit --message "..."` は Docker フォーマット指定時のみ有効 → `--message` を削除して解決
- `trap cleanup EXIT` が先にコンテナを消してしまいユーザーが手動 commit しようとして失敗 → スクリプト内で commit するよう修正

## 8. ~/sniper シンボリックリンクの永続化（systemd サービス）

### 目的
- Podman イメージ内の `/root/sniper` を `~/sniper` として永続マウントし、Claude からも参照できるようにする

### 実装
- `/usr/local/bin/sniper-mount`：Podman イメージの UpperDir を取得し `~/sniper` にシンボリックリンクを貼るスクリプト
- `/etc/systemd/system/sniper-mount.service`：起動時に自動実行される oneshot サービス
- `runuser -u hiragahama --` を使用（systemd 内で sudo なしにユーザー権限の Podman を操作）

### 注意
- rootless Podman はユーザー名前空間を使うため、root からの bind mount は不可 → **シンボリックリンク方式**を採用
- overlay mount はレイヤーパスが長すぎてカーネルの mount オプション上限（4096B）を超える → こちらも断念
- `rebuild_sift_o2.sh` 実行後はイメージの UpperDir が変わるため `sudo systemctl restart sniper-mount.service` が必要

## 12. Xeon E5-2680 v4 サーバの検討と断念

研究室の Xeon サーバ（E5-2680 v4 × 2 = 56 スレッド、Docker 環境）を長時間実験に使えないか検討。

### スループット比較

| | Ultra 9 285K（現行） | Xeon E5-2680 v4 × 2 |
|---|---|---|
| 物理スレッド | 24 | 56 |
| 12TH 同時実行数 | 2本 | 4本 |
| シングルスレッド速度 | 約 2.5× 速い | 基準 |
| 実効スループット | 2 × 2.5 = **5** | 4 × 1.0 = **4** |

→ Xeon は 285K より **25% 遅い**。

### 発生した障害（参考）
1. VSCode Server の prerequisites 未満でリモート開発不可
2. SSH が公開鍵認証のみ → 鍵生成・登録が必要
3. 鍵の base64 部分が短すぎて無効（44文字、正常は68文字）
4. `yuri` ユーザーが docker グループ未所属 → `docker load` で Permission denied

セットアップコストが高く、かつ性能的に劣るため **断念**。285K + `--no-timeout` で放置する方針に確定。
