## 1. NUMA トポロジ（Arrow Lake モデル）

```
Node 0                          Node 1
├─ P-core: CPU 0–3  (4.0 GHz)  ├─ P-core: CPU  8–11 (4.0 GHz)
└─ E-core: CPU 4–7  (3.5 GHz)  └─ E-core: CPU 12–15 (3.5 GHz)
         ↕ DRAM 51.2 GB/s               ↕ DRAM 51.2 GB/s
         ← ローカルレイテンシ 60 ns →
         ← リモートレイテンシ (bus モデル、帯域 102.4 GB/s 共有) →
```

> **注意**: 当初 E-core の設定周波数が `1.0 GHz`（誤り）になっていた。
> 正しくは `3.5 GHz`。この誤りは E-core 戦略（EPO/Scatter）の `sim_seconds` を
> 約 3.5 倍過大評価させており、ML ラベルの信頼性に影響する可能性がある。
> `config/generate_config.py` の `E_FREQ` を修正後、sizeS の再収集が必要。

## 2. 4 戦略の定義

| 戦略 | 説明 | 対象コア |
|---|---|---|
| **Packed** | 全スレッドを Node0 に集約 | Node0 P+E をスレッド順に埋める |
| **Scatter** | スレッドを Node0/Node1 に交互分散 | 0→8→1→9→4→12→… |
| **HPO** | P-core 優先配置 | Node0 P → Node1 P → Node0 E → Node1 E |
| **EPO** | E-core 優先配置 | Node0 E → Node1 E → Node0 P → Node1 P |

### オーケストレータの並列制御

```python
_HOST_CORES = 32   # 仮想コア予算（実コア 24 に対し意図的に過大設定）
_SNIPER_OVERHEAD = 0

# スレッド数別の同時実行数
# 2TH → 16, 4TH → 8, 8TH → 4, 16TH → 2
workers = _HOST_CORES // (num_threads + _SNIPER_OVERHEAD)
```

**投入順序**: `for st in strategies for wl in workloads`（戦略優先）
→ 同一ワークロードの 4 戦略が同時に走らないようにして CPU 競合を回避。

**タイムアウト・リトライ**（2026年6月30日追加）:
- `timeout = max(expected_wallTime × 3, 600)` 秒でプロセスを強制終了
- 最大 1 回リトライ（出力ディレクトリをクリアして再実行）

## 5. Sniper vs gem5 比較

### シミュレーション速度（Class S）

| スレッド数 | gem5 平均 | Sniper 平均 | 高速化比 |
|---|---|---|---|
| 2TH | ~370 s | ~55 s | **~7×** |
| 4TH | ~630 s | ~75 s | **~8×**（BT 除く） |
| 8TH | ~1,010 s | ~35 s | **~29×** |
| 16TH | ~2,500 s | ~24 s | **~104×** |

- lavaMD_S_16: gem5=4024s vs Sniper=5s → **774×**（規則的メモリ gem5 が苦手）
- PR_S_16: gem5=1818s vs Sniper=14s → **130×**

### 最適戦略分布の違い

| | gem5 (Class S) | Sniper (Class S) |
|---|---|---|
| 支配的戦略 | **Packed** (大多数) | **Scatter** (8TH 全13本) |
| HPO 最適 | 4TH で頻出 | **ほぼ出現しない** |
| EPO 最適 | 散発的 | **16TH グラフ系で出現** |

> gem5 と Sniper で戦略ランキングが大きく異なる。
> 要因として (1) E-core 周波数誤設定、(2) gem5 の帯域幅モデルの差異 が疑われる。

## 6. ML パイプライン

### 特徴量（22 次元）

```
基本性能 : avg_ipc, ipc_cv, pe_ratio
メモリ比率: local_ratio, remote_ratio, l2_intensity,
             l3_intensity, mem_intensity, read_ratio
CPI 内訳  : cpi_l3_ratio, cpi_dram_local_ratio, cpi_dram_remote_ratio,
             cpi_branch_ratio, cpi_sync_ratio
μop 混合  : uop_load_ratio, uop_store_ratio, uop_fp_ratio
TLB/分岐  : dtlb_miss_rate, branch_mispredict_rate
DRAM 詳細 : avg_dram_latency, dram_queue_ratio
キャッシュ: l1d_hit_ratio
ALLTH のみ: num_threads（+1次元）
```

プロファイリング実行は **Packed** を基準とする
（全スレッドを Node0 に集約した状態が特徴量の baseline として安定）。

### 評価手法

- **Leave-One-Out CV**（サンプル数 13/スレッド、52/ALLTH）
- **2% タイバレランス**: 複数戦略が最良比 2% 以内 → Strategies 順の先頭を採用
- **評価指標**: accuracy（分類精度）+ perf_ratio（最良比、1.0 = 最適）

### ALLTH 結果（sim_seconds、Class S）

| 手法 | accuracy | perf_ratio |
|---|---|---|
| DT (depth=4) | 80.8% | 1.130 |
| Random Forest | 88.5% | — |
| SVM | — | — |
| Optuna-Hier (BBO) | **90.4%** | **1.019** |

Optuna が発見した最良設計:
- L1: **Scatter** か否か (RF, n_est=50, depth=6)
- L2: Packed / EPO を識別 (RF, n_est=300, depth=6)

残り誤分類 5 件: BC/PR/SSSP_2TH（Packed→Scatter）、lavaMD_8TH（Scatter→Packed）、BFS_16TH（Packed→EPO）

### 特徴量重要度（上位）

| 順位 | 特徴量 | 意味 |
|---|---|---|
| 1 | `pe_ratio` | P/E コア効率差 → 異種コア選択の主軸 |
| 2 | `mem_intensity` | 命令あたり DRAM アクセス → Scatter か否かの判断 |
| 3 | `l2_intensity` | L2 圧力 → キャッシュ局所性 |
| 4 | `local_ratio` | ローカル DRAM アクセス率 |

## 7. 主な技術的問題と解決策

### 7.1 E-core 周波数誤設定
- **問題**: `E_FREQ = 1.0 GHz`（正: ~3.5 GHz）
- **影響**: EPO/Scatter の sim_seconds が 3.5× 過大評価 → ML ラベル汚染の疑い
- **対応**: `config/generate_config.py` を修正後、sizeS 全再実行が必要

### 7.2 ALLTH 特徴量重要度のインデックスバグ
- **問題**: `random_forest.py` が FEATURE_COLS (9次元) 固定のインデックスを使用、ALLTH (10次元) で ValueError
- **解決**: `X.columns.tolist()` で動的取得

### 7.3 BT/HPO タイムアウト
- **問題**: BT/HPO_4TH が他のすべての HPO より 4× 以上遅延（25 分超）
- **真因**: sizeW の SP コンテナ（GOMP_CPU_AFFINITY=0 1）と BT/HPO（=0 1 2 3）が CPU 0-1 で競合。BT は最重ワークロードのため影響が顕在化
- **解決**: タイムアウト（expected×3, min 600s）+ 1 回リトライ実装

### 7.4 4TH 実行時間の異常値
- **問題**: 旧データで BT_S_4=912s（BT_S_2=183s, BT_S_8=363s と比べ不自然）
- **真因**: 投入順が `for wl for st`（ワークロード優先）だったため 4 戦略の BT が全部同時に走り競合
- **解決**: 投入順を `for st for wl`（戦略優先）に変更 → 同一ワークロードを分散

### 7.5 POSM/Regression の CSV 読み込みエラー
- **問題**: RF が `dataset_*.csv` に best_strategy 列を含めずに保存していた
- **解決**: RF 側で `X.copy(); ds["best_strategy"] = y` として正しく保存

## 9. 判明した主な知見

### Sniper における戦略傾向

| スレッド数 | 支配的最適戦略 | 傾向 |
|---|---|---|
| 2TH | Packed (混在あり) | 少スレッドはローカル集中が有利 |
| 4TH | Packed | P-core のみで十分 |
| 8TH | **Scatter 独占** (全 13 本) | P+E 両方使用時は分散が有利 |
| 16TH | Packed (多数) + EPO (グラフ系) | グラフ系は E-core 分散が効く |

### gem5 との主な乖離点

1. **gem5 では HPO が頻出**、Sniper ではほぼゼロ → 帯域幅モデルの差異か E-core 周波数誤設定の影響
2. **gem5 8TH は Packed/HPO 混在**、Sniper 8TH は全 Scatter → NUMA リモートレイテンシモデルの違い
3. lavaMD のシミュレーション時間差が最大（774×）→ gem5 はキャッシュラインレベルのトレースで規則的アクセスが重い
